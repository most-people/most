### 3.5 零知识存储证明

#### 3.5.1 证明方案选择

采用 **Proof-of-Spacetime (PoSt)** 变体：

1. **Challenge-Response 模式**：
   - 合约随机生成挑战（随机偏移量 + 长度）
   - 节点读取对应位置数据
   - 计算 Merkle root 并提交
   - 合约验证 root 与承诺一致

2. **不泄露内容**：
   - 证明只涉及 Merkle tree 路径
   - 不传输实际数据内容

#### 3.5.2 证明流程

```
合约                        节点
  │                          │
  ├── ① 随机挑战 ───────────→│
  │   (random offset, size)  │
  │                          ├── ② 读取数据块
  │                          ├── ③ 计算 Merkle root
  │←── ④ 提交证明 ───────────┤
  │   (merkleRoot, signature)│
  ├── ⑤ 验证 ────────────────│
  │   ✓ 通过 / ✗ 失败        │
  │                          │
  ├── ⑥ 结算收益 / 惩罚 ────→│
```

### 3.6 动态定价引擎

#### 3.6.1 定价公式

```
基础价格 = 存储量(GB) × 时长(天) × 单价(GB/天)

单价 = 基准单价 × 供需系数 × 节点等级系数

供需系数 = 全网总需求存储量 / 全网总可用存储量
  - > 1.0: 资源紧张，价格上涨
  - < 1.0: 资源充足，价格下降

节点等级系数:
  - Lv1 (新节点): 1.0
  - Lv2 (信誉 > 80): 0.95 (5% 折扣吸引订单)
  - Lv3 (信誉 > 95): 0.90 (10% 折扣)
  - Lv4 (顶级节点): 0.85 (15% 折扣)
```

#### 3.6.2 实现位置

```javascript
// server/src/core/pricing.js
export function calculatePrice(options) {
  const { size, duration, redundancy, networkStats } = options

  const basePrice = size * duration * BASE_RATE

  const supplyDemandRatio = networkStats.totalDemand / networkStats.totalSupply
  const demandFactor = Math.max(0.5, Math.min(2.0, supplyDemandRatio))

  const nodeLevelFactor = getNodeLevelFactor(options.nodeReputation)

  return basePrice * demandFactor * nodeLevelFactor
}

## 四、Base 链 + ERC-4337 集成设计

### 4.1 账户系统：mostWallet 确定性钱包

**用户账户完全基于现有 `mostWallet.js` 派生，不引入外部钱包或社交登录。**

```
username + password
    ↓ PBKDF2 (100,000 次迭代)
32 bytes seed (danger)
    ↓ SHA256 + BIP-39
助记词 (mnemonic)
    ↓ BIP-44 (m/44'/60'/0'/0/0)
EVM 地址 (Base 链)
    ↓
合约交互 / 签名交易
```

**核心优势**：
- 用户只需记住 username + password，无需助记词/私钥
- 确定性派生：同一凭据永远生成同一钱包
- 密钥本地派生，不上传服务器
- 兼容 EVM 生态，可直接与 Base 链合约交互

**安全注意**：`mostWallet.js` 的 PBKDF2 迭代次数已设置为 50,000 次（使用 sha512），低端设备约 200ms，安全性与性能平衡。

### 4.2 链选型：Base + ERC-4337 Paymaster

**选择 Base 的理由**：
- Gas 极低（~$0.01/笔），项目方可承担代付成本
- ERC-4337 Paymaster 生态成熟，代付 Gas 实现简单
- Coinbase 生态协同，未来 UX 优化空间大
- EVM 兼容，ethers.js 无缝对接

**代付 Gas 架构**：

```
用户操作 (username + password)
    ↓
mostWallet 派生签名者
    ↓
交易提交至 Bundler
    ↓
Paymaster 验证并代付 Gas (项目方预存 ETH)
    ↓
Base 链执行合约
```

用户完全无感，无需持有 ETH，无需理解 Gas 概念。

  const mnemonic = mostMnemonic(danger)
  const wallet = ethers.HDNodeWallet.fromPhrase(mnemonic).connect(provider)

/**

  return userOp

    return userOp
  }

  return { createOrder, claimRewards }
}
```

## 七、安全考量

### 7.1 加密安全

| 风险 | 应对 |
|---|---|
| 密钥泄露 | 密钥仅存用户设备，不上传；支持密码派生 |
| 加密算法弱点 | AES-256-GCM 经 NIST 认证；PBKDF2 迭代 50,000 次 |
| 随机数弱点 | 使用 `crypto.randomBytes` CSPRNG |

### 7.2 合约安全

| 风险 | 应对 |
|---|---|
| 重入攻击 | OpenZeppelin ReentrancyGuard |
| 整数溢出 | Solidity 0.8+ 自动检查 |
| 权限滥用 | Ownable + 角色分离 |
| 预言机操纵 | 多源价格聚合 |

### 7.3 P2P 安全

| 风险 | 应对 |
|---|---|
| 恶意节点 | 质押惩罚 + 信誉系统 |
| Sybil 攻击 | 质押门槛 + 工作量证明 |
| 数据篡改 | CID 完整性校验 + 多副本交叉验证 |

## 十、关键决策记录

### 10.1 为什么选择 Base + ERC-4337？

| 因素 | Base | Arbitrum | Polygon |
|---|---|---|---|
| Gas 成本 | ~$0.01 | ~$0.01-0.1 | ~$0.001 |
| Paymaster 支持 | ✅ 成熟 | ✅ 支持 | ✅ 支持 |
| 用户门槛 | 极低（Coinbase 生态） | 低 | 中 |
| 去中心化 | 中 | 中高 | 中 |
| 生态成熟度 | 高 | 最高 | 高 |

**决策**：Base + ERC-4337 Paymaster，用户通过 mostWallet 派生钱包交互，项目方代付 Gas，用户无感。

### 10.2 为什么用 mostWallet 而非外部钱包？

- 用户只需记住 username + password，无需助记词/私钥
- 确定性派生，同一凭据永远生成同一钱包
- 密钥本地派生，不依赖第三方
- 与现有身份系统完全兼容

### 10.2 为什么用 AES-256-GCM 而非 ChaCha20？

- AES-NI 硬件加速广泛支持
- GCM 提供认证加密（防篡改）
- Node.js `crypto` 原生支持，无额外依赖

### 10.3 纠删码库选择

| 库 | 语言 | 性能 | 维护状态 |
|---|---|---|---|
| reed-solomon-js | JS | 中 | 活跃 |
| @noble/hashes | JS | 高 | 活跃 |
| WASM RS | Rust→WASM | 极高 | 活跃 |

**决策**：先用 `reed-solomon-js` 快速验证，性能瓶颈时换 WASM。
