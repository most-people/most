| 问题       | 表现                                   |
| ---------- | -------------------------------------- |
| 数据控制权 | 平台可随时删除、审查、限制访问         |

**缺乏纯 P2P 原生、内容寻址、人人可参与、可竞价续费的公共内容保种网络。**

## 二、核心价值主张

| 价值             | 实现方式                                                             |
| ---------------- | -------------------------------------------------------------------- |
| **抗审查**       | 无中心服务器，内容寻址（CID），数据分布在独立节点                     |
| **完全去中心化** | 无中心服务器，订单/质押/审计/结算由智能合约执行                       |
| **零门槛参与**   | 浏览器即可存储文件，普通电脑可部署节点赚取收益                       |
| **内容寻址**     | CID 即权限——类似 IPFS/磁力链接，知道 CID 即可定位节点、下载文件、续费 |
| **市场竞价**     | 用户发布需求，节点报价报名；用户可按价格、信誉、副本数选择节点        |
| **无感体验**     | Base + ERC-4337 Paymaster 代付 Gas，用户无需持有 ETH                 |
| **稳定支付**     | 稳定币（USDC/USDT）支付，无价格波动风险                              |

MostBox 不是隐私网盘。默认文件作为公共内容保种，完整副本置于节点；私密内容必须由上传者在本地加密后再进入 MostBox 流程。

## 三、技术架构

### 3.1 总体流程

文件按 256KB chunk 构建一棵 Merkle 树，链上仅存 `chunkMerkleRoot` 和 `chunkSize`。Hyperswarm topic 取 `cid.multihash.digest`（32 字节）。完整流程见 5.6 存储订单模型。

### 3.2 CID 生成规则

MostBox v1 的 CID 使用现有 `ipfs-unixfs-importer` 生成 UnixFS CID v1。当前显式参数：

| 参数 | 值 |
|------|----|
| `cidVersion` | `1` |
| `rawLeaves` | `true` |
| `wrapWithDirectory` | `false` |
| source path | 固定为 `file`，但不进入根 CID |

`package-lock.json` 当前锁定 `ipfs-unixfs-importer@16.1.5`。未显式写出的 importer 参数（如默认 chunker、DAG layout、hash 算法）不得在协议中静默漂移；升级 importer 前必须先跑黄金样本测试。若黄金样本输出变化，必须视作协议身份变更，单独设计迁移或版本策略。

黄金样本：

| 样本 | size | 期望 CID |
|------|------|----------|
| empty | 0 | `bafkreihdwdcefgh4dqkjv67uzcmw7ojee6xedzdetojuzjevtenxquvyku` |
| `hello world` | 11 | `bafkreifzjut3te2nhyekklss27nh3k72ysco7y32koao5eei66wof36n5e` |
| `{"hello":"world"}` | 17 | `bafkreietui4xdkiu4xvmx4fi2jivjtndbhb4drzpxomrjvd4mdz4w2avra` |
| pattern-300KiB | 307200 | `bafybeicqqpsudfblbjer72m5vzragvjxnpa7tv65nbz72epjrcxqm3jcs4` |

### 3.3 副本模型（m>=2 完整副本）

| 参数 | 含义 |
|------|------|
| m>=2 | 用户选择至少 2 个独立节点，各自持有完整文件副本 |
| 扩容系数 | m 倍（总存储 = 原文 × m） |
| 容错 | 任一节点存活即可下载 |
| 单节点持有 | 完整文件 |
| 下载来源 | 任意 1 个节点 |

完整副本方案无需编码依赖、只需一棵 Merkle 树、下载连 1 个节点即可、修复从存活节点全量拷贝。复杂度远低于纠删码，适合作为 MVP 主线。质押惩罚与赏金审计提供经济安全边际。

### 3.4 节点发现与下载

仅凭 CID 即可完成全部操作：

```
下载者（仅知 CID）
  │
  ├→ 链上查询 OrderFactory.getActiveOrders(cid)
  │     → 选择一个有效 orderId
  │     → order.nodes[m]（ETH 地址列表）
  │     → order.chunkMerkleRoot, order.m
  │
  ├→ topic = cid.multihash.digest（sha2-256 的 32 字节 digest）
  │    Hyperswarm.join(topic)   // 存储节点 + 下载者临时种子聚合在此
  │
  ├→ 发起 ETH 签名挑战验证授权
  │     → 比对 order.nodes[] → 确认是合法存储节点
  │
  └→ 从任意 1 个已验证节点（或任何持有副本的 peer）下载完整文件
        → 本地构建 Merkle 树 → 比对链上 chunkMerkleRoot ✓
        → 重算 UnixFS CID v1(文件) === CID ✓
        → 下载完成后保持连接，作为临时种子服务其他下载者
```

Topic 使用 `cid.multihash.digest`，sha2-256 CID 天然产 32 字节，无需额外 hash 或截断。节点重启时需重新 join 所有持有的 CID。

### 3.5 存储审计（赏金猎人模型）

任何网络参与者都可以审计任何订单的任意存储节点，**仅欺诈上链**。

```
赏金猎人                                  被审计节点
  │                                          │
  │── 1. join(topic) → 连接目标节点              │
  │── 2. 请求 chunk[i] + Merkle proof        │
  │                                          │
  │←─ {chunkData, proof[], nodeSignature} ───│
  │                                          │
  │ 3. 本地验证：                              │
  │    hash(chunk) =? leaf[i]                 │
  │    root(leaf[i], proof[]) =? chunkMerkleRoot │
  │                                          │
  │ 4a. 通过 → 无链上操作（零 gas）              │
  │ 4b. 验证失败 → 提交 fraud proof 上链       │
```

**Fraud proof 最小上链内容**：`{orderId, accusedNode, challengeNonce, chunkIndex, chunkHash, proof[], nodeSignature}`。`chunkData` 仅在链下返回给赏金猎人用于本地验证，不上链。合约执行 `verify(nodeSignature) → root(chunkHash, proof[]) → 比对订单 chunkMerkleRoot`。

**节点签名内容**：`nodeSignature` 必须覆盖 `chainId, orderId, accusedNode, challengeNonce, chunkIndex, chunkHash, proofHash, deadline`。如果节点签名过的 `chunkHash + proof[]` 无法还原到订单 `chunkMerkleRoot`，则视为节点提交了可归责的错误响应。

| 场景 | 合约执行结果 | 后续 |
|------|-------------|------|
| chunk + proof 有效 | fraud claim 为假 | 罚没赏金猎人质押 → 30% 归被诬告节点 |
| chunkHash + proof 无效且节点签名有效 | fraud claim 成立 | 按最低罚没额/比例/重复违规系数罚没节点质押 → 30% 归赏金猎人 → 触发 repair |
| 单次节点不响应 | 无可验证 fraud proof | 扣当期可结算收益/信誉分 |
| 多次独立审计不响应 | 达到治理阈值 | 冻结收益并触发 repair；repair 证实丢失后按最低罚没额/比例/重复违规系数罚没 |

**Counter-proof 反证**：只用于反驳伪造举报，例如 `nodeSignature` 无效、challenge 过期、`orderId/chunkIndex/challengeNonce` 不匹配。若 fraud proof 已包含有效节点签名，且签名覆盖的 `chunkHash + proof[]` 无法还原到链上 `chunkMerkleRoot`，节点不能再用之后提交的正确 chunk 洗白。

**博弈**：
- 节点作弊 → 任何赏金猎人可举报得 30% 罚没 → 作弊期望收益 < 0
- 赏金猎人伪造或错配举报 → 节点提交 counter-proof 反证 → 猎人被反杀
- 无人审计 → 没有 fraud proof 就视为诚实 → 到期正常结算

### 3.6 数据修复

```
检测：赏金猎人提交有效 fraud proof 上链
  │
  ├── 合约标记 node[i] = failed
  ├── 按最低罚没额/比例/重复违规系数罚没质押（30% → 赏金猎人，70% → 销毁）
  ├── 从候选节点池随机选替补
  ├── 替补 join(topic) → 从存活节点全量拷贝完整文件
  ├── 替补本地构建 Merkle 树 → 提交 chunkMerkleRoot 上链验证
  ├── 替补加入 order.nodes[]
  └── repair 带宽费从 Repair 储备金结算
```

### 3.7 热门文件与下载者自动做种

热门文件被大量下载时，质押节点的上行带宽面临压力。解决方案遵循 B 站 WebRTC P2P 加速原理——**下载者边下边推**，和现有 Hyperswarm 行为完全一致：

```
冷文件（少数下载者）：
  下载者 A ─── pull ──→ 质押节点 X
                             ← 其他下载者直连 X 拉取

热门文件（爆发）：
  下载者 A ─── pull ──→ 质押节点 X
  下载者 B ─── pull ──→ 质押节点 X + 下载者 A（A 已完成，做种中）
  下载者 C ─── pull ──→ 质押节点 X + A + B（AB 分担流量）
  ...
  下载者 Z ─── pull ──→ X + A + B + ... + Y（100 个临时种子分摊）
```

Hyperswarm 4.x 的 peer 天然既下又传——无需额外开发，只需客户端下载完成后**不主动断连**，继续在 topic 上 serve 一段时间。

临时种子限制（客户端本地配置，不涉及链上）：
- 最多同时做种 N 个文件
- 下载完成后做种 T 小时（如 24h）
- 上传速率上限（如 1MB/s）

效果：质押节点带宽压力被临时种子分摊，热门文件反而下载更快，节点无需因"被下得多"而回避热门内容。

## 四、网络角色

### 4.1 存储用户

- 使用稳定币（USDC/USDT）支付存储费用
- 上传完整文件到 Hyperdrive → 下单 → 等待 pull 确认 → 可离线
- 通过 CID 随时取回；分享 CID 供他人免费下载
- 可自行本地加密后再存，实现访问控制

### 4.2 存储节点（矿工）

- 贡献硬盘空间和带宽，持有完整文件副本
- 监听链上 `OrderCreated` 事件，按价格、容量和信誉报价报名；被选中后 pull 文件并持续提供下载
- 赚取稳定币收益
- 需质押 MOST 作为保证金
- 声明磁盘容量决定可接订单数上限，谎报→审计发现→罚没
- 任何节点可充当"赏金猎人"审计其他节点获利

### 4.3 社区治理

- MOST 持有者投票决定最小副本数、质押量、手续费率、市场参数
- 审核技术升级提案
- 无特殊权限，无法干预数据或运营

## 五、经济模型

### 5.1 双轨经济

| 代币       | 角色          | 用途                           |
| ---------- | ------------- | ------------------------------ |
| **稳定币** | 支付媒介      | 用户支付存储费用，节点获得收益 |
| **MOST**   | 治理+价值捕获 | 节点质押、赏金猎人质押、社区投票、价值增值 |

### 5.2 MOST 代币属性

| 属性     | 值                        |
| -------- | ------------------------- |
| 名称     | MostBox Token             |
| 符号     | MOST                      |
| 总供应量 | 21,000,000 枚（固定上限） |
| 链       | Base（ERC-20）            |

### 5.3 MOST 分配方案

| 用途           | 比例 | 释放规则                 |
| -------------- | ---- | ------------------------ |
| 节点质押激励池 | 40%  | 按有效存储、在线率、审计结果分期释放，禁止一次性发放 |
| 团队           | 10%  | 4 年线性解锁，1 年 cliff |
| 社区生态       | 30%  | 空投/开发者基金/节点招募，按治理提案分批释放 |
| 储备           | 20%  | 流动性、做市、生态储备，时间锁+多签+治理披露 |

MOST 获取成本必须保持足够高：节点激励只随有效存储、在线率和审计结果分期释放；社区与生态发放必须有锁定、解锁和治理披露；项目方与储备不得低价集中抛售，避免破坏质押安全边际和节点预期。

### 5.4 市场竞价与节点收益

MostBox 不采用固定全网单价。用户发布存储需求，节点按自身成本、信誉、带宽和可用容量报价；用户可自主选择便宜节点、可靠节点，或支付更高费用让更多节点持有副本。

#### 用户侧

| 参数 | 说明 |
|------|------|
| m | 副本数，必须 `m >= 2` |
| duration | 预付周期，默认 1 个月 |
| budget | 用户愿意支付的总预算 |
| preference | 便宜优先 / 信誉优先 / 多地区 / 多副本 / 指定节点 |

#### 节点侧

节点报价单位建议为 `USDC/GB/月`，订单最终价格由用户选择的 m 个节点报价之和决定。

| 节点 | 报价 | 信誉 | 适合场景 |
|------|------|------|----------|
| A | $0.02/GB/月 | 高 | 可靠优先 |
| B | $0.01/GB/月 | 中 | 便宜优先 |
| C | $0.05/GB/月 | 高 + 高带宽 | 热门内容 |

网络只定义交易规则、最低副本数、质押要求和审计惩罚，不强制定价。

#### 网络手续费分配（订单成交价的 10%）

| 用途 | 比例 |
|------|------|
| MOST 回购销毁 | 70% |
| Repair 储备金 | 30% |

举报收益不从手续费出——来自罚没质押（30% 归赏金猎人）。

### 5.5 节点质押

| 参数 | 值 |
|------|-----|
| 质押要求 | 100 MOST / TB 宣称容量（治理可按市场情况调整） |
| 节点收益 | 取决于节点报价、用户选择和有效存储量 |
| 发放原则 | MOST 严格控制释放，减少非必要流通抛压 |
| 作恶罚没 | `max(最低罚没额, 节点总质押 × 订单文件大小 / 节点声明容量 × 重复违规系数)` → 30% 赏金猎人 + 70% 销毁 |
| 赏金猎人质押 | 需质押少量 MOST 方可提交 fraud proof（防止女巫攻击） |

### 5.6 存储订单模型

```
用户充值 USDC → MostBox 钱包
      ↓
文件构建 Merkle 树 → 写入 Hyperdrive → join(topic)
      ↓
链上下单：{cid, chunkMerkleRoot, chunkSize, m>=2, budget, duration}
      ↓
emit OrderCreated 事件 → 所有在线节点监听到
      ↓
节点提交报价/容量/信誉承诺 → 用户选择或按规则抽签
      ↓
选中节点 join(topic) → pull 完整文件 → 验证 chunkMerkleRoot → submit pullConfirm 上链
      ↓
订单生效 → 上传者离线，节点按月领取 USDC
      ↓
到期 → 赏金猎人无 fraud proof → USDC 释放给节点 → 合约标记过期 → 节点释放存储
```

#### 规则汇总

| 特性         | 说明 |
| ------------ | ---- |
| **预付费钱包** | 余额可转出，已下单费用不可退款 |
| **CID 即权限** | 知道 CID 即可查链上节点 → join topic → 下载 |
| **多订单索引** | `orderId` 是主键，同一 CID 可存在多个有效订单 |
| **开放续费** | 任何人可为任意 CID 续费（addFunds，延长 expiresAt） |
| **节点分配** | 报名窗口 + 用户选择 / 自动抽签，m 必须 >= 2；不按响应速度直接分配 |
| **Pull 确认** | 选中节点 pull 完整文件 → 验证 chunkMerkleRoot → submit pullConfirm |
| **Pull 超时** | 下单后 24h 内未满 m 个确认 → 自动退款，订单取消 |
| **预付周期** | 每次 1 个月；到期前可 addFunds 续期 |
| **到期处理** | expiresAt 到达后若无 fraud proof，USDC 释放给节点，合约标记过期 |

#### 订单状态机

```
Created
  → Bidding（节点报价报名）
  → Selected（用户选择或自动抽签）
  → Pulling（选中节点拉取完整文件）
  → Active（满 m 个节点确认，订单生效）
  → Expired（到期无争议，USDC 释放）

异常路径：
  Bidding/Selected/Pulling 超时 → Cancelled + 退款
  Active 期间 fraud proof 成立 → Repairing
  Repairing 成功 → Active
  Repairing 失败且副本数不足 → Degraded / Cancelled（按治理规则处理剩余资金）
```

### 5.7 MOST 价值捕获

| 机制           | 规则 |
| -------------- | ---- |
| **手续费销毁** | 手续费 70% 从 DEX 买入 MOST 并销毁 |
| **罚没销毁**   | 罚没质押 70% 销毁（30% 归赏金猎人） |
| **长期效应**   | 网络使用越多 → 销毁越多 → MOST 越稀缺 → 质押价值越高 |

### 5.8 正向循环

```
用户支付 USDC → 节点获得 90% 稳定币收入
                     ↓
               10% 手续费 → 70% 买入并销毁 MOST
                     ↓
               赏金猎人获罚没 30% → 激励审计
                     ↓
               MOST 严格控发 + 回购销毁 → 获取成本提高，价值支撑增强
                     ↓
               节点质押 MOST 更有价值 → 更多节点加入
                     ↓
               网络容量增加 → 更多用户 → 更多销毁
```

## 六、智能合约架构

| 合约 | 职责 |
|------|------|
| **OrderFactory** | 创建/管理订单，托管 USDC |
| **NodeRegistry** | 节点注册、质押 MOST、信誉分 |
| **FraudProofModule** | 接收 fraud proof → 验证 Merkle 路径 → 执行罚没 |
| **SettlementModule** | 到期无争议释放 USDC |
| **RepairModule** | 节点失败 → 分配替补 → 结算 repair 费 |
| **MOSTToken** (ERC-20) | 治理代币 |
| **MOSTBurner** | 手续费 → DEX 买入 MOST → 销毁 |

**链上与离链分工**：

| 链上（智能合约） | 离链（节点客户端） |
|-----------------|-------------------|
| 订单创建与 USDC 托管 | Merkle 树构建 |
| 节点注册与 MOST 质押 | P2P 文件拉取与上传 |
| chunkMerkleRoot 存储（验证锚） | Hyperswarm topic 加入与发现 |
| Fraud proof Merkle 验证 | 赏金猎人 audit 挑战-响应 |
| 罚没与举报奖励分配 | 下载完整性校验（CID + Merkle） |
| 到期结算与 USDC 释放 | Repair 全量拷贝 |

## 七、竞品对比

| 项目        | 去中心化程度 | 存储方知情权 | 节点门槛      | 支付方式      |
| ----------- | ------------ | ------------ | ------------- | ------------- |
| Filecoin    | 链上去中心化 | 可见哈希     | 高（专业矿机） | FIL           |
| Storj       | 依赖 Satellite | 零知情权     | 中            | USDC/信用卡   |
| Sia         | 链上去中心化 | 可见分片信息 | 中            | Siacoin       |
| **MostBox** | **纯 P2P 原生** | **公开可取；可本地加密** | **极低（普通电脑）** | **USDC+MOST** |

> MostBox 不默认加密文件内容，定位为公共内容保种网络。如需隐私保护，上传者必须自行本地加密后上传密文。

### 差异化优势

1. **纯 P2P 原生** — 无 Satellite、无协调层，CID digest 即 Hyperswarm topic，直连即下载
2. **极简架构** — 无纠删码，无分片，完整文件副本，报名竞价后选中节点存储
3. **内容寻址** — CID 即权限，无需账号体系，无需密钥分发
4. **稳定支付** — 稳定币支付，无价格波动风险
5. **开放续费** — 任何人可为任何文件续费，同一订单追加 USDC
6. **经济安全** — 质押罚没 + 赏金猎人审计，仅欺诈上链，日常零 gas
7. **下载者做种** — P2P 天然边下边推，热门文件越多人下载越快，节点带宽压力被分摊

## 八、风险与应对

| 风险 | 应对 |
|------|------|
| 节点大规模下线 | m>=2 副本 + 自动 repair + 质押惩罚 |
| 赏金猎人不足（无人审计） | 罚没 30% 强激励；下载者下载时天然审计 |
| 赏金猎人诬告 | 被诬节点提交同 chunk 反证 → 猎人质押被反杀 |
| 稳定币脱锚 | 支持多稳定币（USDC/USDT/DAI） |
| 监管限制 | 明确定位公共内容保种市场；私密内容用户侧加密，节点可按本地策略选择是否承接订单 |
| 早期节点不足 | 储备池补贴 + 社区空投招募 |
| 智能合约漏洞 | 分模块设计 + 第三方审计 + 时间锁升级 |
| CID 泄露 → 未授权下载 | 设计如此（CID 即权限）；自行本地加密可控访问 |
| 节点谎报容量 | 声明容量 > 实际 → 存不下 → 审计无法提供 Merkle proof → 按比例质押罚没 |
| 热门文件耗尽节点上传带宽 | 下载者自动做种（3.7），节点带宽压力被临时种子群分摊 |

## 十一、总结

MostBox 基于 Holepunch 生态的纯 P2P 传输层 + Base 智能合约，构建极简的去中心化付费存储网络。

**技术侧**：一个文件、一棵 Merkle 树、m 个副本、一个 32 字节 Hyperswarm topic（取 CID 的 multihash digest）。无纠删码，无分片，无默认加密。CID 即全部——标识文件、定位节点、校验完整性。节点通过报名竞价进入候选池，用户选择或自动抽签后，选中节点 pull 完整副本。赏金猎人审计代替中心化验证（仅 fraud proof 上链，零日常 gas）。下载者完成即自动做种，热门文件 P2P 分摊节点带宽。作恶节点按比例质押罚没（30% 归举报者），诚实节点签收 USDC。

**经济侧**：稳定币 + MOST 双轨模型，存储价格由节点竞价和用户选择决定。手续费 70% 回购销毁 MOST，按比例质押罚没驱动通缩，MOST 严格控制释放以减少非必要抛压。CID 可自由分享，任何人可免费下载、自行续费。网络使用驱动 MOST 销毁与节点供给增长的良性循环。
