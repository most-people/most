# MostBox 智能合约设计

> 以 `项目计划书.md` 为准：Base 链上负责订单、节点报价、质押、审计、结算和修复；文件传输与 Merkle 计算在链下完成。代币属性与分配见项目计划书。

---

## 一、合约架构

```
contracts/
├── MostToken.sol           # ERC-20 代币（MOST，治理+价值捕获）
├── OrderFactory.sol        # 创建/管理存储订单，托管稳定币
├── NodeRegistry.sol        # 节点注册、MOST 质押、容量声明、信誉分
├── FraudProofModule.sol    # fraud proof 验证与罚没
├── SettlementModule.sol    # 到期结算与 USDC 释放
├── RepairModule.sol        # 节点失败后的替补与 repair 结算
├── MostBurner.sol          # 手续费回购并销毁 MOST
└── Governance.sol          # MOST 持有者社区治理（后续阶段）
```

MVP 不包含固定全网定价、纠删码验证库、x402 或周期性 PoSt 证明合约。

## 二、核心数据结构

### 2.1 StorageOrder

```solidity
struct StorageOrder {
    bytes cid;               // UnixFS CID v1 bytes
    bytes32 topic;           // cid.multihash.digest，用于 Hyperswarm topic
    bytes32 chunkMerkleRoot; // 256KB chunk Merkle root
    uint256 chunkSize;       // 默认 256KB
    uint8 replicaCount;      // 用户选择，必须 m>=2
    uint256 size;            // 原始文件大小
    uint256 expiresAt;       // 到期时间
    address stablecoin;      // USDC/USDT
    uint256 budget;          // 用户预算
    address[] storageNodes;  // 用户选择或抽签选中的 m 个节点
    OrderStatus status;
}
```

### 2.2 NodeInfo

```solidity
struct NodeInfo {
```

### 2.3 FraudProof

```solidity
struct FraudProof {
    bytes32 orderId;
    address accusedNode;
    bytes32 challengeNonce;
    uint256 chunkIndex;
    bytes32 chunkHash;
    bytes32[] chunkMerkleProof;
    bytes nodeSignature;
}
```

## 三、订单流程

```
1. 用户上传文件到 Hyperdrive 并 join topic
2. 用户调用 OrderFactory.createOrder()
3. 合约托管 1 个月稳定币费用
4. emit OrderCreated(cid, topic, chunkMerkleRoot, chunkSize, m, budget)
5. 节点监听事件并提交报价、容量、信誉承诺
6. 用户手动选择或自动策略抽签选出 m 个节点
7. 选中节点 join(topic) 并 pull 完整文件
8. 选中节点 submitPullConfirm(orderId, chunkMerkleRoot)
9. m 个确认者成为该订单存储节点
```

订单状态：

```
Created → Bidding → Selected → Pulling → Active → Expired

Bidding/Selected/Pulling 超时 → Cancelled + 退款
Active 期间 fraud proof 成立 → Repairing
Repairing 成功 → Active
Repairing 失败且副本数不足 → Degraded / Cancelled
```

### 预付与续费

| 规则 | 说明 |
| ---- | ---- |
| 预付周期 | 每次下单预付 1 个月 |
| 续费 | 任意人可调用 `addFunds(orderId, amount)` 延长 `expiresAt` |
| 到期 | 到期后若无 fraud proof，稳定币释放给 m 个节点，订单标记过期 |
| 超时 | 24h 内未满 m 个确认则退款，订单取消 |

## 四、市场竞价与分配

| 分配 | 比例 |
| ---- | ---- |
| 存储节点 | 90% |
| 网络手续费 | 10% |

网络手续费继续拆分：70% MOST 回购销毁，30% Repair 储备金。

举报收益来自被罚没节点的 MOST 质押：30% 赏金猎人，70% 销毁。

## 六、赏金审计与 Fraud Proof

### 离链审计

赏金猎人 join(topic)，请求目标节点返回 chunkData + Merkle proof + nodeSignature，本地验证 chunk hash 与 chunkMerkleRoot。验证失败时提交最小 fraud proof：`chunkHash + proof[] + nodeSignature`，不把 chunkData 上链。

`nodeSignature` 必须覆盖 `chainId, orderId, accusedNode, challengeNonce, chunkIndex, chunkHash, proofHash, deadline`。

### 链上处理

| 场景 | 合约结果 |
| ---- | -------- |
| fraud proof 成立 | 按最低罚没额/比例/重复违规系数罚没节点质押，触发 repair |
| fraud proof 为假 | 罚没赏金猎人质押，奖励被诬告节点 |
| 单次节点不响应 | 扣当期可结算收益/信誉分 |
| 多次独立审计不响应 | 冻结收益并触发 repair；repair 证实丢失后罚没 |
| counter-proof 成立 | 赏金猎人质押被罚，部分归被诬节点 |

**Counter-proof 机制**：只用于反驳伪造举报，例如 `nodeSignature` 无效、challenge 过期、`orderId/chunkIndex/challengeNonce` 不匹配。若 fraud proof 已包含有效节点签名，且签名覆盖的 `chunkHash + proof[]` 无法还原到链上 `chunkMerkleRoot`，节点不能再用之后提交的正确 chunk 洗白。

## 七、修复流程

```
fraud proof 成立
  │
  ├── 标记 failed node，按最低罚没额/比例/重复违规系数罚没 MOST
  ├── 从候选节点池选择替补
  ├── 替补 join(topic)，从存活节点全量拷贝完整文件
  ├── 替补提交 chunkMerkleRoot 确认
  └── 替补加入 order.nodes[]
```

repair 带宽费用从 Repair 储备金结算。

## 十、合约安全

| 风险 | 应对 |
| ---- | ---- |
| 重入攻击 | OpenZeppelin ReentrancyGuard |
| 权限滥用 | 分模块角色控制 + 时间锁升级 |
| 稳定币脱锚 | 支持 USDC/USDT/DAI |
| 诬告攻击 | 赏金猎人质押 + counter-proof |
| 报名女巫 | MOST 质押、容量上限、信誉分 |
| DEX 操纵 | 回购使用滑点限制和 TWAP 保护 |
