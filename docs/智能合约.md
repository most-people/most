# MostBox 智能合约设计

> 以 `项目计划书.md` 为准：Base 链上负责订单、质押、审计、结算和修复；文件传输与 Merkle 计算在链下完成。

---

## 一、合约架构

```
contracts/
├── MostToken.sol           # ERC-20 代币（MOST，治理+价值捕获）
├── OrderFactory.sol        # 创建/管理存储订单，托管稳定币
├── NodeRegistry.sol        # 节点注册、MOST 质押、容量声明、信誉分
├── FraudProofModule.sol    # fraud proof 验证与罚没
├── SettlementModule.sol    # 到期结算与 USDC 释放
├── RepairModule.sol        # 节点失败后的替补与 repair 结算
├── MostBurner.sol          # 手续费回购并销毁 MOST
└── Governance.sol          # MOST 持有者社区治理（后续阶段）
```

MVP 不包含动态定价库、纠删码验证库或周期性 PoSt 证明合约。

## 三、订单模型

### 3.1 Order 数据

```solidity
struct StorageOrder {
    bytes contentCid;        // UnixFS CID v1 bytes
    bytes32 contentDigest;   // rootCid.multihash.digest，用于 topic 与索引
    bytes32 merkleRoot;      // 256KB chunk Merkle root
    uint256 chunkSize;       // 默认 256KB
    uint8 replicaCount;      // MVP 默认 m=2
    uint256 size;            // 原始文件大小
    uint256 expiresAt;       // 到期时间
    address stablecoin;      // USDC/USDT
    uint256 prepaidAmount;   // 预付 1 个月费用
    address[] storageNodes;  // 前 m 个 pullConfirm 节点
    OrderStatus status;
}
```

### 3.2 下单与抢单

```
1. 用户上传文件到 Hyperdrive 并 join topic
2. 用户调用 OrderFactory.createOrder()
3. 合约托管 1 个月稳定币费用
4. 合约 emit OrderCreated(CID, digest, merkleRoot, chunkSize, m)
5. 节点监听事件并 join(topic)
6. 节点 pull 完整文件，链下验证 Merkle root
7. 节点 submitPullConfirm(orderId, merkleRoot)
8. 前 m 个确认者成为该订单存储节点
9. 第 m+1 个及之后被拒绝
```

合约不预先匹配节点，节点通过监听事件 + 先到先得竞争获得订单。

### 3.3 预付与续费

| 规则 | 说明 |
| ---- | ---- |
| 预付周期 | 每次下单预付 1 个月 |
| 续费 | 任意人可调用 `addFunds(orderId, amount)` 延长 `expiresAt` |
| 到期 | 到期后稳定币释放给 m 个节点，订单标记过期 |
| 超时 | 24h 内未满 m 个确认则退款，订单取消 |

## 四、定价与分配

MVP 使用固定定价：`$0.1/GB/月`。

| 分配 | 比例 |
| ---- | ---- |
| 存储节点 | 90% |
| 网络手续费 | 10% |

网络手续费继续拆分：

| 用途 | 比例 |
| ---- | ---- |
| MOST 回购销毁 | 70% |
| Repair 储备金 | 30% |

举报收益不从手续费出，来自被罚没节点的 MOST 质押：30% 归赏金猎人，70% 销毁。

## 六、赏金审计与 Fraud Proof

### 6.1 离链审计

```
赏金猎人
  │
  ├── 查询订单与节点列表
  ├── join(topic)
  ├── 请求目标节点返回 chunk[i] + Merkle proof
  ├── 本地验证 chunk hash 与 merkleRoot
  └── 验证失败时提交 fraud proof
```

### 6.2 链上处理

| 场景 | 合约结果 |
| ---- | -------- |
| fraud proof 成立 | 罚没节点质押，触发 repair |
| fraud proof 为假 | 罚没赏金猎人质押，奖励被诬告节点 |
| 节点不响应 | 不直接上链罚没，记录信誉报告 |

Fraud proof 基础字段：

```solidity
struct FraudProof {
    bytes32 orderId;
    address accusedNode;
    uint256 chunkIndex;
    bytes chunkData;
    bytes32[] merkleProof;
    bytes nodeSignature;
}
```

## 七、修复流程

```
fraud proof 成立
  │
  ├── 标记 failed node
  ├── 罚没 MOST
  ├── 从候选节点池选择替补
  ├── 替补 join(topic)
  ├── 从存活节点全量拷贝完整文件
  ├── 替补提交 merkleRoot 确认
  └── 替补加入 order.nodes[]
```

repair 带宽费用从 Repair 储备金结算。

## 十、合约安全

| 风险 | 应对 |
| ---- | ---- |
| 重入攻击 | OpenZeppelin ReentrancyGuard |
| 权限滥用 | 分模块角色控制 + 时间锁升级 |
| 稳定币脱锚 | 支持 USDC/USDT/DAI |
| 诬告攻击 | 赏金猎人质押 + counter-proof |
| 抢单女巫 | MOST 质押、容量上限、信誉分 |
| DEX 操纵 | 回购使用滑点限制和 TWAP 保护 |
