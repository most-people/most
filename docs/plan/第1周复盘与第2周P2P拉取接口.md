## 第 2 周最小接口

| 能力                  | 建议入口                                                       | 输入                                                                | 输出                                  | 验收                             |
| --------------------- | -------------------------------------------------------------- | ------------------------------------------------------------------- | ------------------------------------- | -------------------------------- |
| 记录节点持有文件      | `POST /api/node/holdings` 或引擎内部 `addHolding(record)`      | `cid`、`size`、`chunkMerkleRoot`、`chunkSize`、`localPath`、`topic` | 持久化后的 holding 记录               | 重启后能读取记录。               |
| 列出节点持有文件      | `GET /api/node/holdings`                                       | 无                                                                  | holding 数组                          | Web 管理台可显示当前做种列表。   |
| 按 CID join topic     | 引擎内部 `joinCidTopic(cid)`                                   | `cid`                                                               | topic join 状态                       | 节点启动后自动 join 已持有 CID。 |
| 从 topic 拉取完整副本 | `POST /api/p2p/pull` 或引擎内部 `pullByCid(cid, expectedRoot)` | `cid`、`chunkMerkleRoot`                                            | `savedPath`、`cid`、`chunkMerkleRoot` | 双节点本地拉取成功并通过校验。   |
| 下载后临时做种        | 复用 `downloadFile` 成功后的 Hyperdrive 写入                   | `cid`、本地文件                                                     | 本机加入可服务状态                    | 第三个节点可从下载者拉取。       |

## 风险与边界

- 当前分享链接必须包含 CID、filename 和 `r`；第 2 周的 P2P 拉取接口也应把 `chunkMerkleRoot` 作为必填校验锚。
- 现有 `publishFile` 每个新 CID 都会等待 Hyperdrive discovery flush，集成测试较慢；第 2 周做多节点测试时要控制用例数量。
- `published-files.json` 现在承担用户文件列表与做种元数据两种职责。第 2 周的节点持有列表建议单独落盘，避免污染桌面文件管理视图。
- 下载路径目前写到 `dataPath/<filename>`。节点模式应明确副本存放目录，避免和用户可见文件混在一起。
