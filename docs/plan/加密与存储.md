# MostBox 存储与可选加密设计

> 以 `项目计划书.md` 为准：MostBox 是公共内容保种市场，不是隐私网盘；MVP 不内置默认加密，私密内容由用户在本地自行加密后再上传。

---

## 一、为什么 MVP 不默认加密

项目计划书选择"CID 即权限"的公开内容寻址模型：

- 知道 CID 即可查询订单、定位节点、下载文件、续费。
- 存储节点持有完整文件副本，便于直接下载、做种、repair 和 Merkle 审计。
- 默认加密会引入密钥分发、恢复、分享、节点证明与 UX 复杂度，不进入当前 MVP 主线。

MostBox MVP 的隐私边界：

| 场景            | 处理方式                                                     |
| --------------- | ------------------------------------------------------------ |
| 公共可分享文件  | 直接上传，CID 可自由传播                                     |
| 私密文件        | 上传者在本地自行加密后，将密文作为普通文件上传               |
| CID 泄露        | 设计上等同于获得下载权限                                     |
| 链上元数据      | CID、订单状态、节点地址、存储周期等元数据默认可见            |
| 文件名/路径隐私 | MVP 不承诺链上或节点侧隐私；前端显示元数据仅用于本地应用体验 |

---

## 二、可选本地加密流程

MostBox 不负责管理密钥，但允许用户把加密后的文件作为输入：

```
用户本地工具 / 第三方加密器
  │
  ├── 原始文件 → 密文文件
  ├── 用户自行保存密钥或口令
  └── 密文文件进入 MostBox 上传流程
        │
        ├── CID = UnixFS CID v1（密文）
        ├── chunkMerkleRoot 覆盖密文 chunk
        └── 节点存储密文完整副本
```

MostBox 对密文没有特殊认知：下载者拿到密文后，必须用上传者另行分享的密钥解密。

## 三、对现有代码的影响

| 文件                     | 修改方向                                                                                           |
| ------------------------ | -------------------------------------------------------------------------------------------------- |
| `server/src/core/cid.js` | 继续作为 UnixFS CID v1 计算入口                                                                    |
| `server/src/index.js`    | 增加 chunkMerkleRoot 构建、topic 下载、临时做种控制                                                |
| `published-files.json`   | 继续维护应用层目录结构和展示元数据                                                                 |
| 前端文件管理             | 增加付费存储状态、订单状态、可选"本地加密后上传"提示                                               |
| 桌面端/daemon            | 作为下单、下载、做种、随机抽查、赏金猎人模式和长期存储节点入口；浏览器只展示 demo 并引导下载客户端 |

不新增默认的 `encryption.js` 或 `sharding.js`。

## 四、当前边界

| 类型            | 处理方式                               |
| --------------- | -------------------------------------- |
| 新 MVP 订单文件 | 同样按完整文件 CID 存储                |
| 本地加密文件    | 作为普通密文文件处理                   |
| 未来内置加密    | 作为新的协议设计，不保留旧格式兼容分支 |
