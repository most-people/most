# MostBox 存储与可选加密设计

> MostBox 是 P2P 分享工具，不是隐私网盘或备份服务。默认文件按公共可分享内容处理。

---

## 一、为什么 MVP 不默认加密

MostBox 选择“CID 即权限”的公开内容寻址模型：

- 知道 `most://` 链接即可 join topic 并尝试下载。
- 每个种子持有完整文件副本，便于直接下载、校验和继续做种。
- 默认加密会引入密钥分发、恢复、分享和 UX 复杂度，不进入当前 MVP 主线。

## 二、隐私边界

| 场景 | 处理方式 |
| ---- | -------- |
| 公共可分享文件 | 直接发布，`most://` 可自由传播 |
| 私密文件 | 用户先本地加密，再把密文作为普通文件发布 |
| 链接泄露 | 设计上等同于获得下载权限 |
| 文件名隐私 | `filename` 在链接中明文展示；需要隐藏时用户应改名或去掉敏感信息 |
| 长期保存 | MostBox 不承诺；用户自己保存重要数据 |

## 三、可选本地加密流程

```
用户本地工具 / 第三方加密器
  │
  ├── 原始文件 → 密文文件
  ├── 用户自行保存密钥或口令
  └── 密文文件进入 MostBox 发布流程
        │
        ├── CID = UnixFS CID v1（密文）
        └── 下载者拿到密文并通过 CID 校验后自行解密
```

MostBox 对密文没有特殊认知。

## 四、对现有代码的影响

| 文件 | 修改方向 |
| ---- | -------- |
| `server/src/core/cid.js` | 继续作为 UnixFS CID v1 计算入口 |
| `server/src/index.js` | 发布、下载、下载后做种、holding 管理 |
| `published-files.json` | 继续维护用户可见文件列表 |
| `node-holdings.json` | 维护做种副本列表 |
| 桌面端/daemon | 作为发布、下载、做种和管理入口 |

不新增默认的 `encryption.js` 或 `sharding.js`。

## 五、笔记工具的例外边界

笔记模块已经内置私密内容和备份加密：

| 能力 | 范围 |
| ---- | ---- |
| 私密笔记 | 使用本地 Web3 登录态加密单篇笔记内容 |
| 笔记云备份 | 上传加密后的笔记应用数据到备份 API |
| 本地导入导出 | 导出和导入加密笔记备份文本 |

这些能力只覆盖 `app/note` 的应用数据，不是 MostBox 文件分享协议的默认加密，也不表示 MostBox 对用户分享文件提供云端备份、托管或永久可用承诺。
