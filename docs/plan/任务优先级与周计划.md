# MostBox 任务优先级与周计划

> 目标：像游戏闯关一样推进工程。每一关都有主线任务、支线任务、验收条件和解锁下一关的标准。开发主线优先，但从第 1 周开始并行做轻量运营验证，避免做出没人用的产品。

---

## 一、优先级规则

任务优先级按“能否更快验证产品闭环”排序：

| 优先级 | 判断标准               | 示例                                          |
| ------ | ---------------------- | --------------------------------------------- |
| P0     | 没有它，MVP 闭环跑不通 | Merkle 校验、topic 下载、节点持久化、订单状态 |
| P1     | MVP 可演示但不够可信   | 节点签名审计、单副本警告、订单日志、薄 CLI    |
| P2     | 提升体验或增长验证     | 节点运营面板、引导文案、早期用户访谈          |
| P3     | v1 后再做              | 赏金猎人市场、DAO、多稳定币、自动 repair、ZK  |

当前最重要的主线是：

```
现有 P2P 文件分享
  → 加上 Merkle 校验
  → 加上节点保种模式
  → 加上本地订单模拟
  → 加上测试网 USDT 订单
  → 加上最小审计
  → 形成可演示 MVP
```

## 四、运营与推广要不要提前做

要提前做，但不要一开始就大规模推广。

原因很简单：MostBox 的风险不是“没人知道”，而是“知道的人来了以后不知道该怎么用、为什么要用、敢不敢当节点”。所以运营要从第 1 周开始，但目标是验证定位和招募种子用户，不是追求声量。

### 运营优先级

| 阶段        | 目标         | 要做                                                     | 不做                    |
| ----------- | ------------ | -------------------------------------------------------- | ----------------------- |
| 第 1-2 周   | 验证定位     | 访谈 5-10 个潜在用户                                     | 不投广告、不发大范围 PR |
| 第 3-4 周   | 招募节点候选 | 找 3-5 个愿意跑 Ubuntu/Docker 或 Windows Server 节点的人 | 不承诺收益              |
| 第 5-8 周   | 预热技术社区 | 发布技术进展、演示 GIF、开发日志                         | 不喊“主网上线”          |
| 第 9-10 周  | Alpha 招募   | 邀请小群用户跑真实文件                                   | 不追求增长曲线          |
| 第 11-12 周 | 复盘转化     | 统计下载成功率、节点留存、用户困惑                       | 不扩到不可控规模        |

### 从第一天就要验证的问题

| 问题                          | 验证方式                                                 |
| ----------------------------- | -------------------------------------------------------- |
| 谁真的需要公共内容保种？      | 访谈开源数据集、独立创作者、研究资料分享者、P2P 社区用户 |
| 他们为什么不用现有网盘/IPFS？ | 记录真实痛点，不用替用户脑补                             |
| 他们是否愿意付费？            | 询问愿意为 10GB/月、100GB/月付多少钱                     |
| 谁愿意跑节点？                | 询问硬盘、带宽、在线时间、最低收益预期                   |
| 他们会用什么环境跑节点？      | 区分家用电脑、Ubuntu VPS/独服、Windows Server、NAS       |
| 最大阻力是什么？              | 钱包、USDT、合规、使用复杂度、下载速度逐项记录           |

### 早期定位文案

先用一句话验证：

> MostBox 是一个公共内容保种网络：你为一个 CID 付费，独立节点帮你长期保留完整副本，任何人都能凭 CID 下载并校验文件。

如果用户问“这是不是隐私网盘”，回答必须一致：

> 不是。MostBox 默认面向公共内容；私密文件需要你先本地加密，再把密文上传。

## 五、每周固定节奏

| 时间      | 开发动作                   | 运营动作                    |
| --------- | -------------------------- | --------------------------- |
| 周一      | 定本周主线目标和验收条件   | 确定本周要问的 1 个市场问题 |
| 周二-周四 | 写代码、补测试、跑本地演示 | 约访谈或收集反馈            |
| 周五      | 冻结本周演示，修阻塞 bug   | 发一条开发日志或小范围更新  |
| 周六      | 长测、整理文档             | 整理用户问题列表            |
| 周日      | 复盘：是否解锁下一关       | 判断定位/文案是否要改       |

每周只问一个关键问题，避免运营变成噪音。比如：

- 第 1 周：用户是否理解“公共内容保种”？
- 第 2 周：节点是否愿意为了 USDT 收益长期在线？
- 第 3 周：Ubuntu/Docker 节点安装是否够简单？
- 第 4 周：用户是否愿意手动选择节点？

## 六、当前第一步

从第 1 周 Day 1 开始：

1. 梳理现有上传/下载/CID/Hyperdrive 链路。
2. 写 CID 黄金样本测试。
3. 写 Merkle 工具测试。

第一周不要碰合约。先把文件身份和完整性校验打牢，这是一切订单、审计和节点收益的地基。
