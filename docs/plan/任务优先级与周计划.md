# MostBox 任务优先级与当前执行计划

> 本文是当前任务板。它不再按旧时间表推进，而是按 MVP 候选收尾和 Alpha 验证来排优先级。

## 一、当前目标

目标：把已经跑通的 P2P 做种闭环打磨成可交给小范围用户试用的 MVP 候选版。

成功标准：

1. A 发布文件，B 下载并自动做种，A 退出后 C 能从 B 下载并通过 CID 校验。
2. 应用和 daemon 重启后，已持有 CID 自动恢复 join topic。
3. 管理台能解释本机状态、容量、holding、topic 状态和最近日志。
4. 主应用文案不让用户误以为 MostBox 是云盘、备份服务或付费保种市场。
5. Web3 工具箱仍可独立使用，但不出现在文件分享主流程里。

## 三、优先级规则

| 优先级 | 判断标准 | 当前例子 |
| ------ | -------- | -------- |
| P0 | 没有它就不能交付 MVP 候选 | Quick Start、主流程文案、管理台安全门槛、端到端验收脚本 |
| P1 | MVP 能用，但影响诊断或小范围试用 | 种子健康度、同名文件冲突提示、日志筛选、导出诊断 |
| P2 | Alpha 后再增强体验 | 单文件暂停/恢复、分享历史、导入导出、桌面打包优化 |
| P3 | 只保留评估，不进入近期计划 | `magnet:` 兼容、浏览器长期做种、内置应用层加密 |

| 子任务 | 验收 |

### [ ] 2. 主应用分享和下载体验复查

| 子任务 | 验收 |
| ------ | ---- |
| 发布成功文案 | 明确“本机在线时可下载，下载者完成后会继续做种” |

### [ ] 3. 管理台与 daemon 收尾

| 子任务 | 验收 |
| ------ | ---- |
| 管理台安全策略 | 默认本机可访问，或公网暴露时必须有明确口令/警告 |
| 状态解释 | holding 状态能解释 queued、joining、active、paused、error |

### [ ] 4. Alpha 前长测

| 子任务 | 验收 |
| ------ | ---- |
| 100MB 文件 | 发布、下载、校验、下载后做种全通过 |
| 1GB 文件 | 下载和做种稳定，失败时错误可读 |
| 重启恢复 | daemon 重启后自动 join 已持有 CID topic |
| 发布者退出 | 至少一个下载者在线时，新下载者仍可完成下载 |

## 五、运营验证问题

| 阶段 | 只问一个问题 |
| ---- | ------------ |
| MVP 候选 | 用户是否理解“发链接 + 在线做种 + 下载者接力”？ |
| 小范围 Alpha | 用户是否愿意让下载完成的文件继续做种？ |
| v1 前 | 管理台是否能解释“为什么现在下载不到”？ |

统一定位文案：

> MostBox 是一个 P2P 文件分享工具：你发布文件得到 `most://` 链接，别人下载后默认继续做种，文件靠在线用户传播。

如果用户问“这是不是备份/云盘”，回答必须一致：

> 不是。重要数据请自己保存；MostBox 只负责 P2P 分享和校验，不承诺永久保存。
