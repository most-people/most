## 一、市场痛点

### 1.1 传统网盘

| 问题       | 表现                                   |
| ---------- | -------------------------------------- |
| 数据控制权 | 平台可随时删除、审查、限制访问         |
| 隐私风险   | 服务商可扫描文件内容，存在数据泄露隐患 |
| 体验限制   | 限速、限流、空间付费昂贵               |

### 1.2 现有去中心化存储

| 项目     | 核心缺陷                                             |
| -------- | ---------------------------------------------------- |
| Filecoin | 链上订单+复制证明机制复杂，节点需专业矿机，门槛极高 |
| Storj    | 依赖中心化 Satellite 协调节点发现、审计和结算        |
| Sia      | 自研链生态封闭，分片信息对存储方可见                 |

### 1.3 市场空白

**缺乏纯 P2P 原生、内容寻址、人人可参与、可竞价续费的公共内容保种网络。**

## 二、核心价值主张

| 价值             | 实现方式                                                             |
| ---------------- | -------------------------------------------------------------------- |
| **抗审查**       | 无中心服务器，内容寻址（CID），数据分布在独立节点                     |
| **完全去中心化** | 无中心服务器，订单/质押/审计/结算由智能合约执行                       |
| **零门槛参与**   | 普通电脑运行桌面端/后台节点即可存储文件并赚取收益；浏览器只展示 demo 并引导下载客户端 |
| **内容寻址**     | CID 即权限——类似 IPFS/磁力链接，知道 CID 即可定位节点、下载文件、续费 |
| **市场竞价**     | 用户发布需求，节点报价报名；用户按价格和信誉手动选择 `replicaCount` 个节点 |
| **客户端优先**   | 浏览器只展示 demo；正式下单、下载、做种和审计都在客户端完成           |
| **稳定支付**     | 仅支持 USDT 支付，无原生代币价格波动风险                              |

MostBox 不是隐私网盘。默认文件作为公共内容保种，完整副本置于节点；私密内容必须由上传者在本地加密后再进入 MostBox 流程。

v1 不承诺浏览器常驻做存储节点，也不把浏览器作为正式管理入口。浏览器只展示产品 demo、文档和客户端下载引导；下单、下载、做种、随机抽查和长期存储节点能力都放在桌面端或后台 daemon 中。

## 三、技术架构

### 3.1 总体流程

文件按 256KB chunk 构建一棵 Merkle 树，链上仅存 `chunkMerkleRoot` 和 `chunkSize`。Hyperswarm topic 取 `cid.multihash.digest`（32 字节）。完整流程见 5.4 存储订单模型。

### 3.2 CID 生成规则

MostBox v1 的 CID 使用现有 `ipfs-unixfs-importer` 生成 UnixFS CID v1。当前显式参数：

| 参数 | 值 |
|------|----|
| `cidVersion` | `1` |
| `rawLeaves` | `true` |
| `wrapWithDirectory` | `false` |
| source path | 固定为 `file`，但不进入根 CID |

`package-lock.json` 当前锁定 `ipfs-unixfs-importer@16.1.5`。未显式写出的 importer 参数（如默认 chunker、DAG layout、hash 算法）不得在协议中静默漂移；升级 importer 前必须先跑黄金样本测试。若黄金样本输出变化，必须视作协议身份变更，单独设计迁移或版本策略。

黄金样本：

| 样本 | size | 期望 CID |
|------|------|----------|
| empty | 0 | `bafkreihdwdcefgh4dqkjv67uzcmw7ojee6xedzdetojuzjevtenxquvyku` |
| `hello world` | 11 | `bafkreifzjut3te2nhyekklss27nh3k72ysco7y32koao5eei66wof36n5e` |
| `{"hello":"world"}` | 17 | `bafkreietui4xdkiu4xvmx4fi2jivjtndbhb4drzpxomrjvd4mdz4w2avra` |
| pattern-300KiB | 307200 | `bafybeicqqpsudfblbjer72m5vzragvjxnpa7tv65nbz72epjrcxqm3jcs4` |

### 3.3 副本模型（replicaCount >= 1 完整副本）

| 参数 | 含义 |
|------|------|
| replicaCount >= 1 | 用户手动选择对应数量的独立节点，各自持有完整文件副本 |
| 默认推荐 | `replicaCount = 2`，兼顾成本与基础冗余 |
| 单副本警告 | `replicaCount = 1` 时客户端必须二次确认：无冗余，节点离线或丢失即可能无法下载，手动修复也可能没有可用源 |
| 扩容系数 | `replicaCount` 倍 |
| 容错 | 任一持有完整副本的节点存活即可下载；单副本无冗余 |
| 单节点持有 | 完整文件 |
| 下载来源 | 任意 1 个节点 |

完整副本方案无需编码依赖、只需一棵 Merkle 树、下载连 1 个节点即可、修复从存活节点全量拷贝。复杂度远低于纠删码，适合作为 MVP 主线。质押惩罚与客户端随机抽查提供经济安全边际。

### 3.4 节点发现与下载

仅凭 CID 即可完成全部操作：

```
下载者（仅知 CID）
  │
  ├→ 链上查询 OrderBook.getActiveOrder(cid)
  │     → 获得唯一有效 orderId
  │     → order.nodes[replicaCount]（ETH 地址列表）
  │     → order.chunkMerkleRoot
  │
  ├→ topic = cid.multihash.digest（sha2-256 的 32 字节 digest）
  │    Hyperswarm.join(topic)   // 存储节点 + 下载者临时种子聚合在此
  │
  ├→ 发起 ETH 签名挑战验证授权
  │     → 比对 order.nodes[] → 确认是合法存储节点
  │
  └→ 从任意 1 个已验证节点（或任何持有副本的 peer）下载完整文件
        → 本地构建 Merkle 树 → 比对链上 chunkMerkleRoot ✓
        → 重算 UnixFS CID v1(文件) === CID ✓
        → 下载完成后保持连接，作为临时种子服务其他下载者
```

Topic 使用 `cid.multihash.digest`，sha2-256 CID 天然产 32 字节，无需额外 hash 或截断。节点重启时需重新 join 所有持有的 CID。

**CID 风险边界**：CID 不是访问控制密钥。知道 CID 的任何人都可以查询活跃订单、发现节点并尝试下载；链上也会暴露 CID、订单状态、节点地址和存储周期等元数据。私密内容必须先由用户本地加密，MostBox 只负责保种密文。

### 3.5 存储审计（用户随机抽查 + 赏金猎人）

每个安装客户端的用户都可以参与随机抽查。客户端默认只对自己创建、续费或下载过的订单随机选择 `orderId + node + chunkIndex` 发起挑战。日常通过不产生链上操作；只有可验证欺诈，或达到不响应阈值后的状态变更上链。

同时保留赏金猎人模式：任何客户端都可以主动抽查任意公开订单，并在提交有效 fraud proof 后获得奖励。赏金猎人与普通用户客户端共用同一套挑战-响应和 fraud proof 协议，不单独设计复杂角色系统。

```
审计客户端                                被审计节点
  │                                          │
  │── 1. join(topic) → 连接目标节点              │
  │── 2. 请求 chunk[i] + Merkle proof        │
  │                                          │
  │←─ {chunkData, proof[], nodeSignature} ───│
  │                                          │
  │ 3. 本地验证：                              │
  │    hash(chunk) =? leaf[i]                 │
  │    root(leaf[i], proof[]) =? chunkMerkleRoot │
  │                                          │
  │ 4a. 通过 → 无链上操作（零 gas）              │
  │ 4b. 验证失败 → 提交 fraud proof 上链       │
```

**Fraud proof 最小上链内容**：`{orderId, accusedNode, challengeNonce, chunkIndex, chunkHash, proof[], nodeSignature}`。`chunkData` 仅在链下返回给审计客户端用于本地验证，不上链。合约执行 `verify(nodeSignature) → root(chunkHash, proof[]) → 比对订单 chunkMerkleRoot`。

**节点签名内容**：`nodeSignature` 必须覆盖 `chainId, orderId, accusedNode, challengeNonce, chunkIndex, chunkHash, proofHash, deadline`。如果节点签名过的 `chunkHash + proof[]` 无法还原到订单 `chunkMerkleRoot`，则视为节点提交了可归责的错误响应。

| 场景 | 合约执行结果 | 后续 |
|------|-------------|------|
| chunk + proof 有效 | fraud claim 为假 | 罚没提交者质押 → 30% 归被诬告节点 |
| chunkHash + proof 无效且节点签名有效 | fraud claim 成立 | 按最低罚没额/比例/重复违规系数罚没节点质押 → 30% 归有效提交者 → 订单进入 Degraded |
| 单次节点不响应 | 无可验证 fraud proof | 扣当期可结算收益/信誉分 |
| 多次独立审计不响应 | 达到协议阈值 | 冻结未结算收益、降低信誉、订单进入 Degraded；不直接罚没本金 |

**不响应处理**：不响应本身不是 fraud proof，不能仅凭超时罚没本金。v1 采用更保守的分层处理：审计者提交带 `deadline` 的挑战记录；节点在窗口内返回签名响应即可清除记录；同一订单、同一节点被多个独立审计者在不同时间窗口记录为不响应后，扣未结算收益、降低信誉，并把订单标记为 `Degraded`。只有节点返回了签名错误 proof，且该响应可验证归责，才进入质押罚没流程。

**Counter-proof 反证**：只用于反驳伪造举报，例如 `nodeSignature` 无效、challenge 过期、`orderId/chunkIndex/challengeNonce` 不匹配。若 fraud proof 已包含有效节点签名，且签名覆盖的 `chunkHash + proof[]` 无法还原到链上 `chunkMerkleRoot`，节点不能再用之后提交的正确 chunk 洗白。

**博弈**：
- 节点作弊 → 用户客户端或赏金猎人提交有效 fraud proof 可得 30% 罚没 → 作弊期望收益 < 0
- 审计者伪造或错配举报 → 节点提交 counter-proof 反证 → 提交者被反杀
- 无人审计 → 没有 fraud proof 就视为诚实 → 到期正常结算

### 3.6 手动修复

```
检测：审计客户端提交有效 fraud proof 上链
  │
  ├── 合约标记 node[i] = failed，订单进入 Degraded
  ├── 按最低罚没额/比例/重复违规系数罚没质押（30% → 有效提交者，70% → Treasury）
  ├── 用户手动选择替补节点
  ├── 替补 join(topic) → 从存活节点或临时种子全量拷贝完整文件
  ├── 替补本地构建 Merkle 树 → 提交 chunkMerkleRoot 上链验证
  ├── 替补加入 order.nodes[]
  └── 订单恢复 Active
```

**Repair 失败路径**：
- 若至少 1 个存活节点或临时种子可用，用户手动选择替补节点从可用 peer 全量拷贝，验证通过后加入 `order.nodes[]`
- 若 topic 内没有可用 peer，订单进入 `Degraded`，继续等待节点恢复或下载者临时种子出现
- 若 `Degraded` 超过协议窗口且副本数不足，订单进入 `Cancelled` 或提前结算，剩余未服务资金按协议规则退回/分配
- 替补节点必须重新锁定 USDT 质押；v1 不做自动 repair 调度和自动带宽费结算

### 3.7 热门文件与下载者自动做种

热门文件被大量下载时，质押节点的上行带宽面临压力。解决方案遵循 B 站 WebRTC P2P 加速原理——**下载者边下边推**，和现有 Hyperswarm 行为完全一致：

```
冷文件（少数下载者）：
  下载者 A ─── pull ──→ 质押节点 X
                             ← 其他下载者直连 X 拉取

热门文件（爆发）：
  下载者 A ─── pull ──→ 质押节点 X
  下载者 B ─── pull ──→ 质押节点 X + 下载者 A（A 已完成，做种中）
  下载者 C ─── pull ──→ 质押节点 X + A + B（AB 分担流量）
  ...
  下载者 Z ─── pull ──→ X + A + B + ... + Y（100 个临时种子分摊）
```

Hyperswarm 4.x 的 peer 天然既下又传——无需额外开发，只需客户端下载完成后**不主动断连**，继续在 topic 上 serve 一段时间。

临时种子限制（客户端本地配置，不涉及链上）：
- 最多同时做种 N 个文件
- 下载完成后做种 T 小时（如 24h）
- 上传速率上限（如 1MB/s）

效果：质押节点带宽压力被临时种子分摊，热门文件反而下载更快，节点无需因"被下得多"而回避热门内容。

## 四、网络角色

### 4.1 存储用户

- 使用 USDT 支付存储费用
- 上传完整文件到 Hyperdrive → 下单 → 等待 pull 确认 → 可离线
- 通过 CID 随时取回；分享 CID 供他人免费下载
- 可自行本地加密后再存，实现访问控制

### 4.2 存储节点（矿工）

- 贡献硬盘空间和带宽，持有完整文件副本
- 监听链上 `OrderCreated` 事件，按价格、容量和信誉报价报名；被选中后 pull 文件并持续提供下载
- 赚取 USDT 收益
- 需质押 USDT 作为保证金
- 声明磁盘容量决定可接订单数上限，谎报→审计发现→罚没
- 可设置本地承接策略（文件大小、类型、价格、黑名单、来源策略），自行承担节点所在地内容合规责任
- 客户端可参与随机抽查；开启赏金猎人模式后可抽查任意公开订单，提交有效 fraud proof 可获得罚没奖励

### 4.3 协议治理（v1 简化）

- v1 先由多签/参数管理员管理质押量、手续费率、市场参数
- 所有参数变更公开记录，后续可逐步迁移到社区治理
- 无特殊权限，无法干预数据或运营

## 五、经济模型

### 5.1 v1 单资产模型

MostBox v1 不发行原生代币，全部经济行为仅使用 USDT 完成。这样可以先验证存储市场本身，避免早期把协议安全和代币价格绑定。

| 场景 | 资产 | 说明 |
|------|------|------|
| 用户支付 | USDT | 支付存储订单费用 |
| 节点收益 | USDT | 节点按订单和在线表现获得收入 |
| 节点质押 | USDT | 节点注册和接单保证金 |
| 审计提交者质押 | USDT | 提交 fraud proof 前锁定少量保证金 |
| 罚没与审计奖励 | USDT | 作恶节点被罚，有效提交者获得奖励 |
| Treasury | USDT | 协议手续费和罚没余额先进入单一 Treasury 地址 |

### 5.2 市场竞价与节点收益

MostBox 不采用固定全网单价。用户发布存储需求，节点按自身成本、信誉、带宽和可用容量报价；用户可自主选择便宜节点、可靠节点，或支付更高费用让更多节点持有副本。

#### 用户侧

| 参数 | 说明 |
|------|------|
| replicaCount | 用户自选，必须 `>= 1`，默认推荐 2；选择 1 时客户端必须警告并二次确认 |
| duration | 预付周期，默认 1 个月 |
| budget | 用户愿意支付的总预算 |
| preference | 便宜优先 / 信誉优先 / 多地区 / 指定节点 |

#### 节点侧

节点报价单位建议为 `USDT/GB/月`，订单最终价格由用户手动选择的 `replicaCount` 个节点报价之和决定。

| 节点 | 报价 | 信誉 | 适合场景 |
|------|------|------|----------|
| A | $0.02/GB/月 | 高 | 可靠优先 |
| B | $0.01/GB/月 | 中 | 便宜优先 |
| C | $0.05/GB/月 | 高 + 高带宽 | 热门内容 |

网络只定义交易规则、副本数下限、质押要求和审计惩罚，不强制定价。

#### 网络手续费（订单成交价的 10%）

| 用途 | 比例 |
|------|------|
| Treasury 地址 | 100% |

审计奖励不从手续费出——来自罚没质押（30% 归有效提交者）。v1 不做复杂金库分账，后续再单独设计具体分账规则。

### 5.3 节点质押

| 参数 | 值 |
|------|-----|
| 质押资产 | USDT |
| 质押要求 | 按声明容量锁定 USDT 保证金，具体参数根据存储成本、带宽成本和 repair 成本调整 |
| 节点收益 | 取决于节点报价、用户选择和有效存储量 |
| 作恶罚没 | 仅针对可归责的签名错误 proof；`max(最低罚没额, 节点总质押 × 订单文件大小 / 节点声明容量 × 重复违规系数)` → 30% 有效提交者 + 70% Treasury |
| 审计提交者质押 | 需质押少量 USDT 方可提交 fraud proof（防止女巫攻击） |

### 5.4 存储订单模型

```
用户充值 USDT → MostBox 钱包
      ↓
文件构建 Merkle 树 → 写入 Hyperdrive → join(topic)
      ↓
链上下单：{cid, chunkMerkleRoot, chunkSize, replicaCount>=1, budget, duration}
      ↓
emit OrderCreated 事件 → 所有在线节点监听到
      ↓
节点提交报价/容量/信誉承诺 → 用户手动选择 replicaCount 个节点
      ↓
选中节点 join(topic) → pull 完整文件 → 验证 chunkMerkleRoot → submit pullConfirm 上链
      ↓
订单生效 → 上传者离线，节点按月领取 USDT
      ↓
到期 → 无有效 fraud proof → USDT 释放给节点 → 合约标记过期 → 节点释放存储
```

#### 规则汇总

| 特性         | 说明 |
| ------------ | ---- |
| **预付费钱包** | 余额可转出，已下单费用不可退款 |
| **CID 即权限** | 知道 CID 即可查链上节点 → join topic → 下载 |
| **单活跃订单** | v1 规定同一 CID 同时只能存在 1 个 active order |
| **开放续费** | 任何人可为任意 CID 续费（addFunds，延长 expiresAt） |
| **节点分配** | 报名窗口 + 用户手动选择 `replicaCount` 个节点；不按响应速度直接分配 |
| **Pull 确认** | 选中节点 pull 完整文件 → 验证 chunkMerkleRoot → submit pullConfirm |
| **Pull 超时** | 下单后 24h 内未满 `replicaCount` 个确认 → 自动退款，订单取消 |
| **预付周期** | 每次 1 个月；到期前可 addFunds 续期 |
| **到期处理** | expiresAt 到达后若无 fraud proof，USDT 释放给节点，合约标记过期 |

#### 订单状态机

```
Created
  → Bidding（节点报价报名）
  → Selected（用户手动选择）
  → Pulling（选中节点拉取完整文件）
  → Active（满 replicaCount 个节点确认，订单生效）
  → Expired（到期无争议，USDT 释放）

异常路径：
  Bidding/Selected/Pulling 超时 → Cancelled + 退款
  Active 期间 fraud proof 成立或多次不响应 → Degraded
  Degraded 手动替换成功 → Active
  Degraded 超时且副本数不足 → Cancelled / 提前结算（按协议规则处理剩余资金）
```

### 5.5 正向循环

```
用户支付 USDT → 节点获得 90% 订单收入
                     ↓
               10% 手续费 → Treasury 地址
                     ↓
               有效 fraud proof 提交者获得罚没 30% → 激励审计
                     ↓
               罚没余额进入 Treasury → 覆盖后续补贴和风险成本
                     ↓
               节点获得稳定 USDT 收益 → 更多节点加入
                     ↓
               网络容量增加 → 更多用户 → 更多订单收入
```

### 5.6 后期代币可选项

MostBox v1 不依赖原生代币。若网络规模增长并确实需要去中心化治理，可在后续版本单独提案引入治理代币，用于参数治理、Treasury 管理或节点信誉增强。该方案不作为 MVP 前置条件，且不影响 USDT 作为支付、质押和结算资产。

## 七、竞品对比

| 项目        | 去中心化程度 | 存储方知情权 | 节点门槛      | 支付方式      |
| ----------- | ------------ | ------------ | ------------- | ------------- |
| Filecoin    | 链上去中心化 | 可见哈希     | 高（专业矿机） | FIL           |
| Storj       | 依赖 Satellite | 零知情权     | 中            | USDC/信用卡   |
| Sia         | 链上去中心化 | 可见分片信息 | 中            | Siacoin       |
| **MostBox** | **纯 P2P 原生** | **公开可取；可本地加密** | **极低（普通电脑）** | **USDT** |

> MostBox 不默认加密文件内容，定位为公共内容保种网络。如需隐私保护，上传者必须自行本地加密后上传密文。

### 差异化优势

1. **纯 P2P 原生** — 无 Satellite、无协调层，CID digest 即 Hyperswarm topic，直连即下载
2. **极简架构** — 无纠删码，无分片，完整文件副本，报名竞价后选中节点存储
3. **内容寻址** — CID 即权限，无需账号体系，无需密钥分发
4. **稳定支付** — 仅支持 USDT 支付，无原生代币价格波动风险
5. **开放续费** — 任何人可为任何文件续费，同一订单追加 USDT
6. **经济安全** — 质押罚没 + 用户客户端随机抽查，日常审计链下完成，异常事件才上链
7. **下载者做种** — P2P 天然边下边推，热门文件越多人下载越快，节点带宽压力被分摊

## 八、风险与应对

| 风险 | 应对 |
|------|------|
| 节点大规模下线 | replicaCount 越高越抗下线；Degraded 状态 + 用户手动替换节点 |
| 用户客户端审计不足 | 客户端后台随机抽查 + 下载时天然校验；赏金猎人模式鼓励更多客户端参与 |
| 审计者诬告 | 被诬节点提交同 chunk 反证 → 提交者质押被反杀 |
| USDT 脱锚或合规风险 | v1 仅支持 USDT 以降低复杂度；后续如有需要，可通过协议升级扩展其他结算资产 |
| 监管限制 | 明确定位公共内容保种市场；私密内容用户侧加密，节点可按本地策略选择是否承接订单 |
| 早期节点不足 | 储备池 USDT 补贴 + 社区推广招募 |
| 智能合约漏洞 | 分模块设计 + 第三方审计 + 时间锁升级 |
| CID 泄露 → 未授权下载 | 设计如此（CID 即权限）；自行本地加密可控访问 |
| 浏览器端无法长期在线 | 浏览器只展示 demo 并引导下载客户端；正式功能使用桌面端或后台 daemon |
| 节点承接非法/侵权内容 | 节点可配置本地承接策略并自行决定是否接单；协议不承诺替节点承担内容合规风险 |
| 节点谎报容量 | 声明容量 > 实际 → 存不下 → 审计无法提供 Merkle proof → 按比例质押罚没 |
| 手动修复无可用源 | 订单进入 Degraded，等待节点恢复或临时种子；单副本订单尤其容易无可用源，超时后按协议规则取消/提前结算 |
| 热门文件耗尽节点上传带宽 | 下载者自动做种（3.7），节点带宽压力被临时种子群分摊 |

## 十一、总结

MostBox 基于 Holepunch 生态的纯 P2P 传输层 + Base 智能合约，构建极简的去中心化付费存储网络。

**技术侧**：一个文件、一棵 Merkle 树、用户自选数量的完整副本（`replicaCount >= 1`）、一个 32 字节 Hyperswarm topic（取 CID 的 multihash digest）。无纠删码，无分片，无默认加密。CID 即全部——标识文件、定位节点、校验完整性。节点通过报名竞价进入候选池，用户手动选择 `replicaCount` 个节点后，选中节点 pull 完整副本；单副本订单必须强警告。用户客户端随机抽查 + 赏金猎人模式代替中心化验证，日常审计链下完成，只有可验证欺诈或不响应阈值事件上链。下载者完成即自动做种，热门文件 P2P 分摊节点带宽。签名错误 proof 按比例质押罚没（30% 归有效提交者），不响应只扣收益/信誉并进入 Degraded，诚实节点签收 USDT。

**经济侧**：v1 仅支持 USDT，不发行原生代币。存储价格由节点竞价和用户手动选择决定，USDT 同时承担支付、节点收益、质押、罚没和审计奖励。手续费和罚没余额先进入单一 Treasury 地址，后续再设计复杂分账。CID 可自由分享，任何人可免费下载、自行续费。网络使用驱动节点获得稳定收入，节点供给增长再提升网络容量。
