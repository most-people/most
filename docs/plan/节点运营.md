# MostBox 节点运营规格

> 以 `项目计划书.md` 为准：v1 不发行原生代币，存储节点贡献硬盘与带宽，质押 USDT，监听链上订单，报价报名，入选后持有完整文件副本并赚取 USDT 收益。

---

## 一、节点角色

存储节点是 MostBox 网络的资源提供者，负责保存完整文件副本并向下载者提供 P2P 上传带宽。v1 长期存储节点以桌面端或后台 daemon 为主；浏览器只展示 demo、文档和客户端下载引导，不承载正式节点能力。

MVP 节点按两种形态设计：

| 形态 | 适合对象 | MVP 定位 |
| ---- | -------- | -------- |
| 桌面端节点 | 个人用户、家用电脑、低门槛体验者 | 用于展示、下载、临时做种、轻量保种和审计 |
| 服务器 daemon 节点 | Ubuntu VPS/独服、Windows Server、长期在线运营者 | 作为稳定供给侧优先支持，通过浏览器 Web 管理台日常运营，是 Alpha 节点招募主力 |

因此节点核心逻辑必须先做成可无头运行的 daemon/API，并内置 Web 管理台。服务器节点日常操作通过浏览器访问；CLI 只保留初始化、安装服务、启动和排障入口。桌面端只是另一种控制台。

### 核心职责

- 质押 USDT 作为保证金，声明可用容量。
- 监听 `OrderCreated` 事件，提交报价、容量和信誉承诺。
- 入选后 join 订单 topic，pull 完整文件，本地验证 chunkMerkleRoot 后提交 `pullConfirm`。
- 订单有效期内持续提供下载。
- 可设置本地承接策略（文件大小、类型、价格、黑名单、来源策略），自行承担节点所在地内容合规责任。
- 普通客户端可随机抽查自己相关订单；开启赏金猎人模式后可抽查任意公开订单并提交有效 fraud proof 获得奖励。

### 收益来源

- 存储订单费用：用户支付 USDT，90% 分给订单存储节点。
- 审计奖励：客户端随机抽查或赏金猎人模式发现欺诈后，有效提交者获得被罚没质押的 30%。
- 早期节点激励：来自 Treasury 的 USDT 补贴。

## 二、节点运营流程

### 2.1 注册与报价

```
部署节点客户端 → 配置存储目录和容量上限 → 质押 USDT → NodeRegistry 注册
  → 接入 Hyperswarm 网络 → 监听 OrderCreated → 对合适订单提交报价
```

### 2.2 订单接收与存储

```
OrderCreated
  │
  ├── 节点读取 {cid, topic, chunkMerkleRoot, chunkSize, replicaCount}
  ├── 提交报价、容量和信誉承诺
  ├── 等待用户手动选择
  ├── 入选后 join(topic) → pull 完整文件 → 本地重算 chunkMerkleRoot
  ├── submitPullConfirm(orderId, chunkMerkleRoot)
  └── 成为订单存储节点
```

合约不按响应速度直接分配节点。节点靠价格、容量、信誉、带宽和质押资格进入候选池，最终由用户手动选择 replicaCount 个节点。若 replicaCount=1，客户端必须提示无冗余风险并二次确认。

### 2.3 收益结算

- 用户每个订单预付 1 个月 USDT 费用。
- 到期后若无 fraud proof，90% 费用释放给已确认存储节点。
- 10% 进入单一 Treasury 地址，v1 不做复杂金库分账。
- 节点返回签名错误 proof 并被 fraud proof 证明后，按最低罚没额/比例/重复违规系数罚没 USDT 质押；30% 归有效提交者，70% 进入 Treasury。
- 单次审计不响应只扣当期可结算收益/信誉分；多次独立审计不响应则冻结未结算收益并进入 Degraded，不因超时直接罚没本金。

## 三、信誉分

信誉分不负责订单匹配，只用于展示、协议参数和风控参考。

```
信誉分 = 基础分 100
  - fraud proof 成立: -50 分
  - 单次不响应审计: -5 分/次
  - 多次独立审计不响应: 冻结未结算收益并进入 Degraded
  - Degraded 超时未恢复: -20 分
  + 连续在线 30 天: +5 分
  + 订单到期正常结算: +1 分/单
```

## 四、节点部署

| 形态 | 平台 | 交付物 | 优先级 |
| ---- | ---- | ------ | ------ |
| Linux daemon | Ubuntu 22.04/24.04 LTS | systemd service + Web 管理台 + 最小 CLI | P0 |
| Docker 节点 | Ubuntu/Debian/CentOS 等服务器 | Docker image + Web 管理台 + volume 挂载 | P0 |
| Windows Server daemon | Windows Server 2019/2022 | Windows Service + Web 管理台 + 最小 CLI | P1 |
| 桌面端节点 | Windows/macOS/Linux 桌面 | Electron 控制台 | P1 |

MVP Alpha 优先招募 Ubuntu/Docker 节点，因为它们更接近真实长期在线供给。Windows Server 需要进入计划，但可以在 Ubuntu daemon 稳定后补齐服务安装和运维脚本。

部署完成后统一访问 `http://server-ip:1976/admin`。具体安装命令放到后续 Quick Start，计划文档只保留交付目标，避免提前维护多套未实现命令。

### Web 管理台范围

| 页面 | MVP 功能 |
| ---- | -------- |
| 概览 | 节点在线状态、版本、节点 ID、监听地址、已用/可用容量 |
| 存储 | 已持有 CID、文件大小、`chunkMerkleRoot`、做种状态 |
| 订单 | 待报价、已报价、已入选、Active、Degraded |
| 策略 | 容量上限、最低报价、最大文件大小、是否自动报价 |
| 日志 | join、pull、verify、serve、order event、error |
| 钱包 | 测试网地址、质押状态、收益概览 |

Web 管理台必须默认只监听本机或要求设置访问口令；服务器公网暴露时需要明确安全提示。MVP 不做复杂 RBAC。

### API 与 AI 入口

节点所有管理能力先提供 HTTP API 和 WebSocket 事件，并生成 OpenAPI 文档。Web 管理台、桌面端、薄 CLI 和未来 MCP/Agent 工具都复用同一套 API。

| 入口 | 范围 |
| ---- | ---- |
| HTTP API | 配置、状态、订单、存储、审计、钱包 |
| WebSocket | 节点状态、订单状态、下载/做种进度、错误事件 |
| OpenAPI 文档 | 给 AI、脚本和第三方集成使用 |
| 薄 CLI | 初始化、安装服务、启动/停止、打印诊断信息 |
| MCP/Agent 工具 | MVP 后考虑，直接暴露常用节点操作给 AI |

### 系统要求

| 资源 | 最低要求        | 推荐配置    |
| ---- | --------------- | ----------- |
| CPU  | 2 核            | 4 核        |
| 内存 | 4 GB            | 8 GB        |
| 存储 | 100 GB SSD      | 1 TB NVMe   |
| 带宽 | 10 Mbps         | 100 Mbps    |
| 网络 | 公网 IP 或 UPnP | 固定公网 IP |

## 五、P2P 安全

| 风险 | 应对 |
| ---- | ---- |
| 恶意节点 | USDT 质押 + fraud proof 最低罚没额/比例/重复违规系数 |
| Sybil 攻击 | 质押门槛 + 容量上限 + 信誉分 |
| 数据篡改 | CID 完整性校验 + chunkMerkleRoot |
| 谎报容量 | 存不下订单会在审计/下载中暴露并触发罚没 |
| 非法/侵权内容 | 节点本地承接策略 + 自行决定是否接单 |
| 热门文件带宽压力 | 下载者完成后临时做种，分摊上传压力 |
