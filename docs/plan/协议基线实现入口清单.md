# 协议基线实现入口清单

> 对应 `任务优先级与周计划.md` Day 1：梳理 `server/src/core/cid.js`、上传/下载链路、数据目录写入点。不改业务。

## CID 入口

| 入口                             | 位置                     | 现状                                                                                                                                                                     | 后续任务                                                  |
| -------------------------------- | ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------- |
| `calculateCid(content, options)` | `server/src/core/cid.js` | 使用 `ipfs-unixfs-importer` 生成 UnixFS CID v1，默认 `rawLeaves: true`、`cidVersion: 1`、`wrapWithDirectory: false`；字符串输入走文件流，Buffer 输入走 `Readable.from`。 | Day 2 给空文件、小文件、跨 chunk 文件补黄金样本测试。     |
| `validateCidString(cidString)`   | `server/src/core/cid.js` | 只做非空字符串与 CID v1 `b` 前缀检查。                                                                                                                                   | Merkle/topic 接入前保持现状，后续如需更严格解析再单独改。 |
| `parseMostLink(link)`            | `server/src/core/cid.js` | 只接受 `most://<cid>?filename=...&r=...`。                                                                                                                               | 下载链路继续复用，避免新增链接解析入口。                  |

## 上传链路

| 步骤            | 入口                                                     | 位置                  | 写入/输出                                                                                               |
| --------------- | -------------------------------------------------------- | --------------------- | ------------------------------------------------------------------------------------------------------- |
| 接收 multipart  | `POST /api/publish`                                      | `server/index.js`     | `parseMultipartBusboy` 先落临时文件，调用完成后 `fs.unlink` 删除临时文件。                              |
| 发布文件        | `engine.publishFile(result.filePath, result.filename)`   | `server/src/index.js` | 校验路径与大小，调用 `calculateCid`，用 CID digest 派生 `drive-${hashHex}`。                            |
| 写入 Hyperdrive | `drive.createWriteStream('/' + cidString)`               | `server/src/index.js` | Hyperdrive key 固定为 `/<cid>`，文件显示路径不进入 Hyperdrive。                                         |
| 写入元数据      | `#publishedFiles.push(...)` → `#savePublishedMetadata()` | `server/src/index.js` | 写入 `dataPath/published-files.json`，字段含 `fileName`、`cid`、`driveName`、`publishedAt`、`starred`。 |
| 返回客户端      | `publishResult`                                          | `server/src/index.js` | 返回 `cid`、`most://<cid>?filename=...&r=...`、`fileName`、`chunkMerkleRoot`。                          |

## 下载链路

| 步骤         | 入口                                                     | 位置                  | 写入/输出                                                                                          |
| ------------ | -------------------------------------------------------- | --------------------- | -------------------------------------------------------------------------------------------------- |
| 请求下载     | `POST /api/download`                                     | `server/index.js`     | 先解析链接并检查本地是否已有 CID；异步调用 `engine.downloadFile`，状态通过 WebSocket 推送。        |
| 查找 drive   | `engine.downloadFile(link, taskId)`                      | `server/src/index.js` | `parseMostLink` → `CID.parse` → digest 派生 `drive-${hashHex}`，以 server/client 模式 join swarm。 |
| 拉取内容     | `#waitForDriveContent(drive, ...)`                       | `server/src/index.js` | 轮询 `drive.list()` 与 `drive.update()`，找到 Hyperdrive 条目后开始读取。                          |
| 写入本地文件 | `fs.createWriteStream(savePath)`                         | `server/src/index.js` | 当前 `savePath` 是 `dataPath/<sanitizedFileName>`。                                                |
| CID 校验     | `calculateCid(savePath)`                                 | `server/src/index.js` | 当前只比对下载文件 CID 的 multihash digest，不做 Merkle 校验。                                     |
| 下载后做种   | `drive.createWriteStream('/' + cidString)`               | `server/src/index.js` | 若 drive 内没有 `/<cid>`，把下载到本地的文件再写入 Hyperdrive。                                    |
| 写入元数据   | `#publishedFiles.push(...)` → `#savePublishedMetadata()` | `server/src/index.js` | 下载成功后同样写入 `dataPath/published-files.json`，让本机可预览/分享/继续做种。                   |

## 已发布文件读取链路

| 入口                               | 位置                  | 现状                                                                             |
| ---------------------------------- | --------------------- | -------------------------------------------------------------------------------- |
| `GET /api/files`                   | `server/index.js`     | 返回 `engine.listPublishedFiles()`，来自 `published-files.json` 内存副本。       |
| `GET /api/files/:cid/download`     | `server/index.js`     | 校验 CID 字符串后调用 `engine.readFileRaw(cid, { offset, limit })`，支持 Range。 |
| `engine.readFileRaw(cid, options)` | `server/src/index.js` | 通过 `published-files.json` 记录找到 `driveName`，读取 Hyperdrive key `/<cid>`。 |

## 数据目录写入点

| 路径                            | 写入入口                          | 说明                                                            |
| ------------------------------- | --------------------------------- | --------------------------------------------------------------- |
| `~/.most-box/config.json`       | `server/index.js` 的 `saveConfig` | 保存用户配置；`MOSTBOX_DATA_PATH` 可覆盖数据目录。              |
| `dataPath`                      | `MostBoxEngine.start()`           | 初始化时创建目录，并作为 `Corestore(dataPath)` 的存储根。       |
| `dataPath/published-files.json` | `#savePublishedMetadata()`        | 已发布/已下载文件的显示元数据，是移动、重命名、列表和预览入口。 |
| `dataPath/trash-files.json`     | `#saveTrashMetadata()`            | 回收站元数据。                                                  |
| `dataPath/channels.json`        | `#saveChannelsMetadata()`         | 聊天频道列表。                                                  |
| `dataPath/channel-config.json`  | `#saveChannelConfig()`            | 频道配置。                                                      |
| `dataPath/<sanitizedFileName>`  | `downloadFile()`                  | 下载完成后的本地文件副本。                                      |
| `dataPath` 内 Corestore 文件    | `Corestore` / `Hyperdrive`        | Hyperdrive 内容与复制状态，业务层通过 drive API 写入。          |
| 系统临时目录 `mostbox-uploads`  | `parseMultipartBusboy` / 清理逻辑 | 上传中转文件，不属于持久数据目录，发布后删除。                  |

## Day 2/3 接入边界

- CID 黄金样本测试应直接覆盖 `calculateCid`，不要绕过到上传 API。
- Merkle 工具建议作为新的 `server/src/core/*` 模块接入，先独立单测，再由上传/下载链路调用。
- `published-files.json` 是接入 `chunkMerkleRoot` 的最小元数据位置；Hyperdrive key 仍保持 `/<cid>`，避免破坏移动/重命名零成本原则。
