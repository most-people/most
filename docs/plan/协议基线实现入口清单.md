# 协议基线实现入口清单

> 本文只记录和协议基线直接相关的实现入口：CID、`most://`、发布、下载、holding 和数据目录。

## CID 入口

| 入口                           | 位置                     | 约束                                                    |
| ------------------------------ | ------------------------ | ------------------------------------------------------- |
| `calculateCid(content)`        | `server/src/core/cid.js` | 使用 `ipfs-unixfs-importer` 生成 UnixFS CID v1          |
| `validateCidString(cidString)` | `server/src/core/cid.js` | 校验非空字符串、CID 版本和格式                          |
| `parseMostLink(link)`          | `server/src/core/cid.js` | 接受 `most://<cid>?filename=...`，并保留兼容裸 CID 解析 |

CID 参数固定为：

| 参数                | 值      |
| ------------------- | ------- |
| `cidVersion`        | `1`     |
| `rawLeaves`         | `true`  |
| `wrapWithDirectory` | `false` |

升级 importer、调整 chunk 行为或改 CID 入口前，必须先跑 CID 黄金样本测试。

## 发布链路

| 步骤            | 入口                                    | 写入或输出                              |
| --------------- | --------------------------------------- | --------------------------------------- |
| 接收文件        | `POST /api/publish`                     | multipart 临时落盘，发布后删除临时文件  |
| 发布文件        | `engine.publishFile(content, filename)` | 计算 UnixFS CID v1                      |
| 写入 Hyperdrive | `drive.createWriteStream('/' + cid)`    | Hyperdrive key 固定为 `/<cid>`          |
| 写入文件元数据  | `published-files.json`                  | 用户可见文件名、CID、收藏和目录信息     |
| 写入做种记录    | `node-holdings.json`                    | CID、size、topic、local path、join 状态 |
| join topic      | `joinCidTopic(cid)`                     | topic = `cid.multihash.digest`          |
| 返回客户端      | `publishResult`                         | `cid`、`most://` 链接、文件名           |

## 下载链路

| 步骤         | 入口                                           | 写入或输出                               |
| ------------ | ---------------------------------------------- | ---------------------------------------- |
| 请求下载     | `POST /api/download`                           | 解析 `most://`，异步启动下载任务         |
| 直接拉取     | `POST /api/p2p/pull`                           | 按 CID 或链接拉取完整副本                |
| join topic   | `engine.downloadFile()` / `engine.pullByCid()` | topic = `cid.multihash.digest`           |
| 拉取内容     | `#waitForDriveEntry('/' + cid)`                | 只接受 Hyperdrive 中精确的 `/<cid>` 文件 |
| 写入本地文件 | `fs.createWriteStream(savePath)`               | 保存到本机下载目录                       |
| 完整性校验   | `calculateCid(savePath)`                       | 下载后重算 CID，必须与链接 CID 一致      |
| 下载后做种   | 写入 Hyperdrive + holding                      | 下载者成为新的种子                       |

## 做种持有记录

| 入口                        | 说明                             |
| --------------------------- | -------------------------------- |
| `engine.listHoldings()`     | 列出当前节点持有的可做种副本     |
| `engine.addHolding(record)` | 添加 holding 并 join topic       |
| `engine.pullByCid(input)`   | 按 CID 或 `most://` 拉取完整副本 |
| `engine.joinCidTopic(cid)`  | 按 CID digest 加入文件 topic     |
| `GET /api/node/holdings`    | API 层列出 holding               |
| `POST /api/node/holdings`   | API 层手动添加 holding           |
| `POST /api/p2p/pull`        | API 层触发 CID 拉取              |

holding 记录必须和 `published-files.json` 分开，避免做种副本污染用户文件管理视图。

## 数据目录写入点

| 路径                               | 写入入口                   | 说明                                                |
| ---------------------------------- | -------------------------- | --------------------------------------------------- |
| `~/.most-box/config.json`          | `createNodeConfigStore`    | 保存数据目录、容量和单文件上限                      |
| `dataPath/published-files.json`    | `#savePublishedMetadata()` | 用户可见文件列表                                    |
| `dataPath/node-holdings.json`      | `#saveHoldingsMetadata()`  | 做种副本列表                                        |
| `dataPath/trash-files.json`        | `#saveTrashMetadata()`     | 回收站元数据                                        |
| `dataPath/channels.json`           | `#saveChannelsMetadata()`  | 聊天频道列表                                        |
| `downloadPath/<sanitizedFileName>` | `downloadFile()`           | 下载完成后的本地文件副本，默认 `dataPath/downloads` |
| `dataPath` 内 Corestore 文件       | Corestore / Hyperdrive     | P2P 内容存储                                        |
| 系统临时目录 `mostbox-uploads`     | `parseMultipartBusboy`     | 发布中转文件，发布后删除                            |

## 验证边界

| 场景         | 验证                                                |
| ------------ | --------------------------------------------------- |
| CID 黄金样本 | 空文件、小文件、跨 chunk 文件 CID 固定              |
| 链接解析     | 发布返回的 `most://` 必须包含并解析 CID 和 filename |
| 发布元数据   | 发布后返回 CID、链接，并写入可见文件列表和 holding  |
| 下载校验     | 下载完成后重算 CID，不能跳过内容校验                |
| 下载后做种   | 第三个节点可从下载者拉取                            |
| 重启恢复     | `node-holdings.json` 恢复后自动 join topic          |

优先命令：`npm run test:protocol`。

## 非协议入口

以下入口虽然也使用 CID、加密或 P2P 相关工具，但不属于 MostBox 文件分享协议基线，不能替代本文上面的 UnixFS CID / `most://` / holding 入口：

| 入口                                               | 范围                              | 注意                                            |
| -------------------------------------------------- | --------------------------------- | ----------------------------------------------- |
| `server/src/utils/noteUtils.js#calculateNoteCid()` | 笔记内容和备份 payload 的 raw CID | 不是 UnixFS 文件 CID，不用于 `most://` 文件分享 |
| `server/src/utils/noteBackup.js`                   | 笔记加密备份、恢复和签名认证      | 只覆盖笔记应用数据，不是文件云盘或做种持久化    |
| `server/src/index.js` channel 方法                 | P2P 聊天频道和消息                | 独立工具，不改变文件发布、下载、做种协议        |
