# MostBox MVP 落地计划

> MVP 只验证一件事：文件能靠 `most://` 链接和在线种子传播，下载完成的人默认接力做种。

## 一、MVP 合约

MVP 闭环：

```
发布者选择文件
  -> 生成 UnixFS CID v1
  -> 生成 most:// 分享链接
  -> 发布者按 CID topic 做种
  -> 下载者凭链接下载完整副本
  -> 下载者重算 CID 并校验
  -> 下载者默认持续做种
  -> 发布者退出后，新下载者可从下载者种子继续下载
```

完成定义：

1. 只凭 `most://` 链接能完成发现、下载和 CID 校验。
2. 发布成功和下载成功后都固定自动加入做种列表。
3. 做种记录持久化到 `node-holdings.json`，重启后自动恢复 join topic。
4. 管理台能看到本机正在做种的 CID、大小、topic 状态和基础日志。
5. 发布者退出后，只要至少一个下载者仍在线做种，后续下载者仍能完成下载。

## 二、MVP 范围

| 模块       | MVP 要求                                                        | 验收方式                 |
| ---------- | --------------------------------------------------------------- | ------------------------ |
| CID        | UnixFS CID v1，CID 即完整性锚                                   | 黄金样本测试             |
| 分享链接   | `most://<cid>?filename=...`                                     | 链接解析单测             |
| 发布后做种 | Hyperdrive key 固定为 `/<cid>`，topic 为 `cid.multihash.digest` | 发布后 holding active    |
| 下载校验   | 下载后重算 CID，必须与链接一致                                  | 篡改和错误码测试         |
| 下载后做种 | 校验通过后写入 Hyperdrive 和 holding                            | 第三个节点可从下载者拉取 |
| 做种持久化 | `node-holdings.json` 保存 CID、size、topic、path、状态          | 重启恢复测试             |
| 节点设置   | 数据目录、容量、单文件上限                                      | API 测试 + 手动重启      |
| 管理台     | 状态、容量、持有副本、日志、设置                                | 手动验证                 |
| Alpha 长测 | 真实用户、真实机器、真实文件                                    | 反馈记录和回归验证       |

## 三、明确不做

| 能力                              | MVP 决策                          |
| --------------------------------- | --------------------------------- |
| 上链存储协议                      | 不做                              |
| 智能合约 / USDT / 质押 / 结算     | 不做                              |
| 存储订单 / 报价 / 手动选节点      | 不做                              |
| 赏金猎人 / fraud proof / Treasury | 不做                              |
| 云端上传或托管                    | 不做                              |
| 默认加密                          | 不做；用户自行本地加密            |
| 浏览器长期做种                    | 不承诺；正式能力在桌面端或 daemon |
| `magnet:` 兼容                    | 不进入 MVP，后续单独评估          |
| 全局关闭自动做种                  | 不做；发布和下载成功后默认做种    |
| 应用层限速和同时做种上限          | 不做；后续按真实压力评估          |

笔记、Web3 和 P2P 聊天保留为独立工具，但不作为发布、下载或做种的前置条件。笔记云备份只覆盖笔记应用数据，不计入 MostBox 文件分享 MVP 的可用性或备份承诺。

## 四、用户故事

### 发布者

1. 选择文件。
2. 看到 CID 和 `most://` 链接。
3. 复制链接发给别人。
4. 保持应用在线，让第一批下载者拉到文件。

验收：第一个下载者能凭链接下载并校验。

### 下载者

1. 输入 `most://` 链接。
2. 客户端 join CID topic 并下载。
3. 下载完成后显示校验结果。
4. 默认继续做种。

验收：第二个下载者能从第一个下载者处拉取文件。

### 做种节点 / daemon

1. 配置数据目录、容量和单文件上限。
2. 查看持有副本和 topic join 状态。
3. 后台长期在线，持续为本机持有的 CID 做种。

验收：重启后自动恢复做种列表。

## 五、最终验收剧本

MVP 完成时，应能现场演示：

1. A 发布文件并复制 `most://` 链接。
2. B 输入链接下载，校验通过后自动做种。
3. A 退出。
4. C 输入同一链接，从 B 下载成功并通过 CID 校验。
5. B 重启后，管理台仍能看到该 CID，并自动 join 对应 topic。

## 六、MVP 指标

| 指标                 | 目标                            |
| -------------------- | ------------------------------- |
| 本地三节点传播       | 100% 通过                       |
| 100MB 文件下载后校验 | 重算 CID 一致                   |
| 1GB 文件长测         | 可完成，失败时错误可读          |
| 发布者退出后下载     | 至少 1 个下载者种子在线即可成功 |
| 节点重启恢复         | 自动 join 已持有 CID topic      |
| 节点设置 API         | 不包含订单、USDT、报价字段      |
