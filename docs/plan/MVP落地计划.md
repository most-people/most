# MostBox MVP 落地计划

> 本文用于把 `项目计划书.md` 和 `实施路线图.md` 收敛成可执行 MVP。原则：先验证“用户愿意为公共内容保种付费，节点愿意用普通设备承接并获得收益”，暂不追求完整 v1 协议形态。

---

## 一、MVP 定义

MVP 只做一个最小闭环：

```
上传者选择文件
  → 生成 CID 与 chunkMerkleRoot
  → 本地/测试网创建存储订单
  → 节点报价并被用户手动选择
  → 节点通过 Hyperswarm 拉取完整文件副本
  → 下载者仅凭 CID 下载并校验
  → 到期后订单结算
  → 可用一次最小审计证明节点确实持有文件
```

MVP 的成功标准不是“协议完全去中心化”，而是证明三件事：

1. 文件可以脱离上传者，由选中的节点继续保种。
2. 用户能理解并完成付费、选节点、下载、校验的流程。
3. 节点能按订单获得可解释的 USDT 收益，并承担可测试的失信成本。

---

## 二、MVP 范围

### 必须包含

| 模块 | MVP 要求 | 验收方式 |
| ---- | -------- | -------- |
| CID 与 Merkle | 复用 UnixFS CID v1；按 256KB chunk 生成 `chunkMerkleRoot` | 黄金样本测试 + 篡改检测 |
| P2P 拉取 | topic 使用 `cid.multihash.digest`；节点可从上传者或其他节点拉取完整文件 | 本地 3 节点集成测试 |
| 完整副本 | 不做分片、不做纠删码；每个节点保存完整文件 | 关闭上传者后仍可下载 |
| 手动选节点 | 用户从报价节点中手动选择 `replicaCount` 个节点 | UI 手动测试 |
| 单副本警告 | `replicaCount=1` 必须二次确认 | UI 手动测试 |
| 测试网订单 | 支持创建订单、报价、选择、pullConfirm、到期结算 | Hardhat + 测试网演示 |
| 节点模式 | Ubuntu/Docker daemon + Web 管理台优先，Windows Server daemon 与桌面端控制台跟进 | 节点重启后自动 join 已持有 CID topic |
| API 文档 | HTTP API + WebSocket 事件生成 OpenAPI/事件文档 | Web 管理台、脚本、AI 可复用 |
| 下载校验 | 下载后同时校验 CID 与 `chunkMerkleRoot` | 集成测试 |
| 最小审计 | 支持请求随机 chunk + proof + 节点签名；本地验证 | 单元测试 + 手动演示 |

### 明确不做

| 能力 | MVP 决策 |
| ---- | -------- |
| 原生代币 / DAO | 不做；只使用 USDT 或测试 USDT |
| 多稳定币 | 不做 |
| 默认加密 | 不做；私密文件由用户本地自行加密 |
| 浏览器长期节点 | 不做；浏览器只展示 demo 和下载入口 |
| 自动抽签 | 不做；用户手动选择节点 |
| 自动 repair | 不做；最多标记 Degraded，并让用户手动处理 |
| 复杂信誉系统 | 不做完整算法；只记录基础成功/失败次数 |
| 赏金猎人开放审计 | MVP 可留接口，不作为首版验收项 |
| 图形界面依赖 | 不允许；服务器节点必须可无头运行 |
| 完整 CLI 产品 | 不做；CLI 只保留安装、启动、诊断等薄入口 |
| ZK / PoSt | 不做；使用 Merkle audit |
| 同 CID 多活跃订单 | 不做；同一 CID 同时最多 1 个 active order |
| 复杂 Treasury 分账 | 不做；手续费进入单一地址 |

## 三、落地版本切片

### MVP-0：无链 P2P 保种闭环

目标：先证明现有 P2P 技术栈能可靠完成“上传者离线后仍可下载”。

| 任务 | 产出 | 验证 |
| ---- | ---- | ---- |
| Merkle 工具 | `chunkMerkleRoot`、proof 生成与验证 | 单元测试覆盖空文件、小文件、跨 chunk 文件 |
| CID 黄金样本 | 固定 `ipfs-unixfs-importer` 输出 | 依赖升级前测试必须通过 |
| topic 下载 | 通过 CID digest join topic 并拉取文件 | 本地 3 节点下载成功 |
| 节点持久化 | 节点记录已持有 CID，重启后重新 join | 重启节点后仍可被发现 |
| 临时做种 | 下载完成后保持一段时间上传能力 | 第二个下载者可从临时种子拉取 |

验收标准：

- 上传者 A 上传 1MB、100MB 两个文件后，节点 B/C 可拉取完整副本。
- A 退出后，下载者 D 仍可从 B 或 C 下载并通过 CID + Merkle 校验。
- B 重启后自动重新 join 已持有文件 topic。

### MVP-1：测试网订单闭环

目标：证明订单、节点报价、用户选择、节点确认和结算流程可用。

| 任务 | 产出 | 验证 |
| ---- | ---- | ---- |
| NodeRegistry | 节点注册、质押、容量声明 | Hardhat 单元测试 |
| OrderBook | 创建订单、报价、选择、pullConfirm、结算 | Hardhat 单元测试 |
| 测试 USDT | 本地 mock USDT 或测试网 USDT 配置 | 部署脚本可重复执行 |
| 前端订单 UI | 创建订单、查看报价、选择节点、查看状态 | 手动从上传到 Active |
| 节点监听器 | 监听 `OrderCreated` 并报价 | 本地节点能自动发现订单 |
| pullConfirm | 节点拉取并校验后确认 | 未选中节点确认会失败 |

验收标准：

- 用户创建订单后，至少 2 个节点能报价。
- 用户手动选择 `replicaCount=2` 后，两个节点完成 pullConfirm，订单进入 Active。
- 到期结算后，节点可领取收益，手续费进入 Treasury。
- 同一 CID 不能创建第二个 active order。

### MVP-2：最小审计与失败处理

目标：证明节点不是“报了名就完事”，至少能被抽查。

| 任务 | 产出 | 验证 |
| ---- | ---- | ---- |
| 挑战协议 | 请求 `chunkIndex` 对应 chunk + proof | 本地验证通过 |
| 节点签名 | 响应覆盖 orderId、node、chunkHash、proofHash、deadline | 伪造签名失败 |
| 本地审计 UI | 对自己订单发起一次抽查 | 手动审计通过 |
| 错误 proof 演示 | 节点返回错误 proof 时可识别 | 单元测试 |
| Degraded 标记 | 审计失败后订单进入 Degraded | 合约/集成测试 |

验收标准：

- 正常节点能通过随机 chunk 审计。
- 篡改节点返回的 proof 后，本地验证失败。
- 测试环境中可提交一次最小 fraud proof，并让订单进入 Degraded。

## 四、用户故事

### 上传者

1. 选择文件并看到 CID、大小、预计费用。
2. 选择保种月数和 `replicaCount`。
3. 从报价列表中选择节点。
4. 等待节点确认后离线。
5. 复制 CID 分享给其他人。

验收：上传者离线后，别人仍能通过 CID 下载文件。

### 下载者

1. 输入 CID。
2. 客户端查询订单并 join topic。
3. 从任意可用 peer 下载。
4. 下载完成后显示 CID 与 Merkle 校验结果。
5. 默认临时做种一段时间。

验收：下载者不需要账号，只凭 CID 完成下载和校验。

### 存储节点

1. 配置数据目录、容量、最低报价。
2. 质押测试 USDT 并注册。
3. 自动监听订单并按策略报价。
4. 入选后自动拉取文件并确认。
5. 到期领取收益。

验收：节点从接单到结算不需要手动拷贝文件。

MVP 节点优先支持 Ubuntu/Docker daemon，并通过浏览器 Web 管理台完成日常运营。AI 与脚本优先使用结构化 API/OpenAPI；CLI 只保留初始化、安装服务、启动服务和诊断信息这类薄入口。桌面端负责低门槛体验和节点控制台，Windows Server daemon 进入 P1，但不能影响 daemon 核心设计。

## 五、关键风险与降级方案

| 风险 | MVP 降级方案 |
| ---- | ------------ |
| Base 测试网 USDT 流程拖慢 | 先用本地 Hardhat + mock USDT 完成演示，再接测试网 |
| 合约审计复杂度过高 | MVP 合约只保留订单、报价、选择、确认、结算；复杂 slashing 后置 |
| P2P NAT 环境不稳定 | MVP 验收先覆盖本地多节点和可控公网节点 |
| 1GB 文件测试耗时 | 日常 CI 用小文件；发布前手动跑 100MB/1GB 长测 |
| 节点审计边界复杂 | 首版只做本地验证和单次 fraud proof 演示，不开放赏金猎人市场 |
| 用户理解成本高 | UI 强调“公共内容保种”和“CID 即可下载”，单副本必须强提示 |

---

## 六、MVP 指标

| 指标 | 目标 |
| ---- | ---- |
| 本地 3 节点文件复制成功率 | 100% |
| 100MB 文件下载后校验 | CID 与 `chunkMerkleRoot` 均一致 |
| 上传者离线后下载 | 至少 1 个存储节点在线即可成功 |
| 节点重启恢复 | 自动 join 已持有 CID topic |
| 订单 Active 成功率 | 测试环境连续 10 单成功 |
| 单副本风险确认 | `replicaCount=1` 无法跳过二次确认 |
| 正常审计 | 随机 chunk proof 本地验证通过 |
| 错误审计 | 篡改 proof 必须失败 |

---

## 七、执行顺序

1. 先做 `MVP-0`，不碰合约：验证 P2P 和校验闭环。
2. 再做 `MVP-1`，用 mock USDT 完成合约与客户端订单闭环。
3. 最后做 `MVP-2`，加入最小审计和 Degraded。
4. MVP 演示通过后，再回到 `实施路线图.md` 的 Phase 3/4 完善赏金猎人、罚没、手动修复和 Treasury。

---

## 八、完成定义

MVP 完成时，应能现场演示：

1. 启动 1 个上传者、2 个存储节点、1 个下载者。
2. 上传者创建订单并选择 2 个节点。
3. 两个节点自动拉取文件并确认。
4. 上传者退出。
5. 下载者输入 CID，成功下载并通过双重校验。
6. 对任意一个节点发起随机 chunk 审计并通过。
7. 篡改一个审计响应，系统能识别失败并把订单标记为 Degraded。

如果这 7 步稳定可重复，MostBox 就具备继续扩展 v1 的落地基础。
