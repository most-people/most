# MostBox 实施路线图

> 以 `项目计划书.md` 为准：先完成 USDT 单资产、用户自选副本数、手动选节点、可续费、可审计的公共内容保种市场；v1 不发行原生代币。

> 执行顺序以 `MVP落地计划.md` 为入口：先完成 MVP-0 无链 P2P 保种闭环，再做 MVP-1 测试网订单闭环，最后补 MVP-2 最小审计与失败处理。本路线图描述完整 v1 主线，不代表所有能力都进入第一版交付。

---

## MVP 优先级

| 切片 | 目标 | 对应阶段 | 是否首版必做 |
| ---- | ---- | -------- | ------------ |
| MVP-0 | CID + Merkle + Hyperswarm 完整副本保种 | Phase 1 | 是 |
| MVP-1 | 测试网订单、报价、手动选节点、pullConfirm、结算 | Phase 2 | 是 |
| MVP-2 | 随机 chunk 审计、节点签名、最小 Degraded | Phase 3/4 的最小子集 | 是 |
| v1 扩展 | 赏金猎人市场、完整罚没、手动修复、Treasury 报表、生态运营 | Phase 3-5 | MVP 后推进 |

首版验收只要求证明“上传者可离线、节点可保种、下载者可校验、订单可结算、节点可被抽查”。完整赏金猎人、复杂信誉、自动化运营和主网生态不阻塞 MVP。

## Phase 1 — 基础数据管道（预计 2 周）

目标：跑通 `Merkle → Hyperswarm 上传/下载 → CID 校验` 全流程，先在本地/模拟环境验证。

| 任务 | 产出 | 验证 |
| ---- | ---- | ---- |
| Merkle 树构建与验证 | 256KB chunk 的 `chunkMerkleRoot` 与 proof 工具 | 单元测试：任意 chunk proof 可验证到 root |
| CID 计算对齐 | 复用现有 UnixFS CID v1 计算 | 黄金样本测试：空文件、小文件、跨 chunk 文件 CID 不漂移 |
| Hyperdrive 完整副本写入 | key 使用 `/<cid>` 存储完整文件 | 集成测试：上传后可按 CID 读取 |
| Hyperswarm topic 下载 | topic 使用 `cid.multihash.digest` | 双节点本地测试：下载节点可 pull 文件 |
| 下载完整性校验 | 下载后重算 chunkMerkleRoot 与 CID | 集成测试：篡改文件必须失败 |

**里程碑**：v0.2.0 — 无链环境下可通过 CID 完成 P2P 上传、下载、校验与临时做种。

**验收标准**：本地启动 3 个节点；覆盖 1MB / 100MB / 1GB 文件；关闭任意 1 个存储节点后仍可下载；节点重启后自动重新 join 已持有 CID 的 topic；CID 黄金样本测试固定。

## Phase 2 — 链上订单（预计 2-3 月）

目标：实现 USDT 托管订单、节点 USDT 质押注册、报价报名、用户手动选择 replicaCount 个节点与到期结算。

| 任务 | 产出 | 验证 |
| ---- | ---- | ---- |
| Hardhat 项目初始化 | `contracts/` + 部署脚本 | `npx hardhat compile` 通过 |
| NodeRegistry | 节点注册、USDT 质押、容量声明 | 单元测试：注册、质押、退出 |
| OrderBook | `{cid, chunkMerkleRoot, chunkSize, replicaCount>=1, duration}` 订单 | 单元测试：创建订单并托管 USDT |
| 报价报名 | 节点提交价格、容量、信誉承诺 | 单元测试：无效报价被拒绝 |
| 手动选择节点 | 用户手动选出 replicaCount 个节点 | 单元测试：必须选择足量有效节点 |
| 单副本警告 | replicaCount=1 时强警告并二次确认 | 手动测试：单副本无法无提示下单 |
| pullConfirm | 选中节点完成 pull 并验证 root 后获得订单 | 并发测试：未选中节点确认被拒绝 |
| 单 CID 单活跃订单 | 同一 CID 同时只有 1 个 active order | 单元测试：重复创建 active order 被拒绝 |
| 订单状态机 | Created/Bidding/Selected/Pulling/Active/Expired/Degraded/Cancelled | 单元测试：超时、退款、降级状态转移 |
| 到期结算 | USDT 释放给已确认节点 | 时间推进测试：到期后可结算 |
| 客户端集成 | 下单、订单状态、节点管理 UI | 手动测试：从上传到订单生效闭环 |

**里程碑**：v0.3.0 — Base 测试网上可创建 USDT 付费订单，节点可报价报名，用户手动选择 replicaCount 个节点，选中节点通过 pullConfirm 生效并到期领取 USDT；单副本订单必须有明确风险确认。

## Phase 3 — 用户随机抽查 + 赏金猎人审计（预计 2-3 月）

目标：普通客户端随机抽查自己相关订单，赏金猎人模式可抽查任意公开订单。日常审计链下完成，只有可验证欺诈或不响应阈值事件上链。

| 任务 | 产出 | 验证 |
| ---- | ---- | ---- |
| Merkle Proof 挑战-响应 | 客户端随机抽查协议 + 赏金猎人模式 | 单元测试：随机 chunk proof 可通过 |
| 节点响应签名 | 节点对审计响应签名 | 单元测试：伪造响应无法通过 |
| FraudProofModule | 最小上链 fraud proof：chunkHash + proof + nodeSignature | 合约测试：签名错误响应成立、好 proof 失败 |
| Counter-proof | 仅反驳签名伪造、挑战过期或字段错配 | 合约测试：有效签名坏响应不能被洗白 |
| 不响应惩罚 | 扣收益/信誉 → Degraded；不因超时直接罚本金 | 集成测试：多次独立不响应触发 Degraded |
| 信誉分记录 | 多人报告、失败、恢复状态记录 | 集成测试：状态转移符合规则 |

**里程碑**：v0.4.0 — 普通客户端和赏金猎人都可审计订单节点；签名错误 proof 上链后触发 USDT 罚没，不响应达到阈值后进入 Degraded。

---

## Phase 4 — 手动修复与简单 Treasury（预计 1-2 月）

目标：节点失败后进入 Degraded，由用户手动选择替补节点；手续费和罚没余额先进入单一 Treasury 地址。

| 任务 | 产出 | 验证 |
| ---- | ---- | ---- |
| Degraded 状态 | 失败节点替换前的降级状态 | 集成测试：fraud proof 后进入 Degraded |
| 手动替换节点 | 用户选择替补节点加入订单 | 集成测试：新节点校验 chunkMerkleRoot 后加入 |
| 全量修复流程 | 替补从存活节点或临时种子拷贝完整文件 | 集成测试：恢复到用户选择的副本数 |
| 简单 Treasury | 手续费和罚没余额进入单一地址 | 合约测试：入账正确 |

**里程碑**：v0.5.0 — 网络可从 Degraded 手动恢复到用户选择的副本数，并完成简单 Treasury 入账。

---

## Phase 5 — 生态与后期治理（预计 2 月）

目标：从测试网验证进入主网与社区运营，公开 Treasury 和节点补贴计划。是否引入治理代币留到后续单独提案。

| 任务 | 产出 | 验证 |
| ---- | ---- | ---- |
| 社区推广 | 节点/开发者/早期用户运营 | 推广计划和反馈记录公开 |
| 节点招募与 USDT 补贴 | 早期节点激励 | 补贴规则和发放记录可审计 |
| Treasury 透明报表 | USDT 收支披露 | 报表与链上记录一致 |
| 后期治理提案 | 是否引入治理代币 | 社区提案单独评估，不作为 v1 前置条件 |
| 公测运营 | 节点招募与用户反馈 | 指标：有效节点数、存储量、成功下载率 |

**里程碑**：v1.0.0 — 完整付费存储网络公测启动。

## 总体进度

```
v0.1.0 (当前)       →  v0.2.0 (数据管道)  →  v0.3.0 (链上订单)
P2P 文件分享            CID/chunkMerkleRoot 校验            USDT 托管
P2P 聊天                topic 下载/做种        USDT 质押
Web3 工具箱             完整副本存储           报价/选择/pullConfirm

v0.4.0 (随机抽查)   →  v0.5.0 (手动修复)  →  v1.0.0 (公测)
Fraud proof             Degraded              生态运营
Counter-proof           手动替换节点           USDT 补贴
Slashing                Treasury 地址          金库报表
```

## 暂不进入主线的能力

| 能力 | 处理方式 |
| ---- | -------- |
| 默认应用层加密 | 用户可自行本地加密；内置加密不作为 MVP 任务 |
| 浏览器正式节点能力 | 浏览器只展示 demo 并引导下载客户端；正式功能使用桌面端/daemon |
| 原生代币 / DAO 治理 | 不作为 v1 任务；后续单独提案 |
| 多稳定币 | 不进入 v1；当前仅支持 USDT |
| 分片与纠删码 | 作为未来 V2 研究，不进入当前路线图 |
| 固定全网定价 | 不采用；MVP 使用节点报价与用户选择 |
| 自动抽签 | 不进入 v1；用户手动选择节点 |
| 自动 repair | 不进入 v1；先做 Degraded + 手动替换 |
| ERC-4337 Paymaster | 不进入 v1；用户自行承担 gas |
| 同 CID 多活跃订单 | 不进入 v1；同一 CID 同时只有 1 个 active order |
| 复杂金库分账 | 不进入 v1；先使用单一 Treasury 地址 |
| x402 | 暂不考虑，后续单独评估 |
| 零知识证明 | MVP 使用 Merkle audit + fraud proof，不做 PoSt/ZK |
