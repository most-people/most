# MostBox 智能合约设计

> 以 `项目计划书.md` 为准：v1 不发行原生代币，Base 链上负责 USDT 订单、节点报价、USDT 质押、审计、结算和手动修复；文件传输与 Merkle 计算在链下完成。

---

## 一、合约架构

```
contracts/
├── OrderBook.sol           # 订单、报价、手动选择、pullConfirm、续费与结算
├── NodeRegistry.sol        # 节点注册、USDT 质押、容量声明、信誉分
└── FraudProofModule.sol    # fraud proof 验证与罚没
```

MVP 不包含原生代币、多稳定币、DAO 治理、自动抽签、自动 repair、ERC-4337 Paymaster、复杂 treasury 分账、固定全网定价、纠删码验证库、x402 或周期性 PoSt 证明合约。手续费和罚没余额先进入单一 Treasury 地址。

## 二、核心数据结构

### 2.1 StorageOrder

```solidity
struct StorageOrder {
    bytes cid;               // UnixFS CID v1 bytes
    bytes32 topic;           // cid.multihash.digest，用于 Hyperswarm topic
    bytes32 chunkMerkleRoot; // 256KB chunk Merkle root
    uint256 chunkSize;       // 默认 256KB
    uint8 replicaCount;      // 用户自选，必须 >= 1
    uint256 size;            // 原始文件大小
    uint256 expiresAt;       // 到期时间
    address usdtToken;       // v1 only USDT
    uint256 budget;          // 用户预算
    address[] storageNodes;  // 用户手动选择的 replicaCount 个节点
    OrderStatus status;
}
```

v1 规定同一 CID 同时只能存在 1 个 active order。续费通过 `addFunds(orderId, amount)` 延长现有订单，不创建并行订单。

### 2.2 NodeInfo

```solidity
struct NodeInfo {
    address nodeAddress;
    uint256 usdtStake;
    uint256 declaredCapacityBytes;
    uint256 usedCapacityBytes;
    uint256 reputation;
    bool active;
}
```

### 2.3 FraudProof

```solidity
struct FraudProof {
    bytes32 orderId;
    address accusedNode;
    bytes32 challengeNonce;
    uint256 chunkIndex;
    bytes32 chunkHash;
    bytes32[] chunkMerkleProof;
    bytes nodeSignature;
}
```

## 三、订单流程

```
1. 用户上传文件到 Hyperdrive 并 join topic
2. 用户调用 OrderBook.createOrder()
3. 合约托管 1 个月 USDT 费用
4. emit OrderCreated(cid, topic, chunkMerkleRoot, chunkSize, replicaCount, budget)
5. 节点监听事件并提交报价、容量、信誉承诺
6. 用户手动选择 replicaCount 个节点
7. 选中节点 join(topic) 并 pull 完整文件
8. 选中节点 submitPullConfirm(orderId, chunkMerkleRoot)
9. replicaCount 个确认者成为该订单存储节点
```

订单状态：

```
Created → Bidding → Selected → Pulling → Active → Expired

Bidding/Selected/Pulling 超时 → Cancelled + 退款
Active 期间 fraud proof 成立或多次不响应 → Degraded
Degraded 手动替换成功 → Active
Degraded 超时且副本数不足 → Cancelled / 提前结算
```

### 预付与续费

| 规则 | 说明 |
| ---- | ---- |
| 预付周期 | 每次下单预付 1 个月 |
| 续费 | 任意人可调用 `addFunds(orderId, amount)` 延长 `expiresAt` |
| 到期 | 到期后若无 fraud proof，USDT 释放给已确认节点，订单标记过期 |
| 超时 | 24h 内未满 replicaCount 个确认则退款，订单取消 |
| 单副本 | replicaCount=1 时客户端必须警告并二次确认：无冗余，节点离线或丢失即可能无法下载 |

| 分配 | 比例 |
| ---- | ---- |
| 存储节点 | 90% |
| 网络手续费 | 10% |

## 五、节点注册与质押

| 参数 | 规则 |
| ---- | ---- |
| 质押资产 | USDT |
| 质押要求 | 按声明容量锁定 USDT 保证金，具体参数根据存储成本、带宽成本和 repair 成本调整 |

| 罚没分配 | 30% 有效提交者，70% Treasury |

## 六、用户随机抽查与 Fraud Proof

### 离链审计

普通用户客户端按本地配置，只随机抽查自己创建、续费或下载过的订单。赏金猎人模式可主动抽查任意公开订单。两者 join(topic) 后请求目标节点返回 chunkData + Merkle proof + nodeSignature，本地验证 chunk hash 与 chunkMerkleRoot。正常响应只在本地验证，不产生链上操作；验证失败时提交最小 fraud proof：`chunkHash + proof[] + nodeSignature`，不把 chunkData 上链。

`nodeSignature` 必须覆盖 `chainId, orderId, accusedNode, challengeNonce, chunkIndex, chunkHash, proofHash, deadline`。

### 链上处理

| 场景 | 合约结果 |
| ---- | -------- |
| fraud proof 成立 | 按最低罚没额/比例/重复违规系数罚没节点质押，订单进入 Degraded |
| fraud proof 为假 | 罚没提交者质押，奖励被诬告节点 |
| 单次节点不响应 | 扣当期可结算收益/信誉分 |
| 多次独立审计不响应 | 冻结未结算收益、降低信誉、订单进入 Degraded；不直接罚没本金 |
| counter-proof 成立 | 提交者质押被罚，部分归被诬节点 |

不响应本身不是 fraud proof，不能仅凭超时罚没本金。只有节点返回了签名错误 proof，且该响应可验证归责，才罚没 USDT 质押。

**Counter-proof 机制**：只用于反驳伪造举报，例如 `nodeSignature` 无效、challenge 过期、`orderId/chunkIndex/challengeNonce` 不匹配。若 fraud proof 已包含有效节点签名，且签名覆盖的 `chunkHash + proof[]` 无法还原到链上 `chunkMerkleRoot`，节点不能再用之后提交的正确 chunk 洗白。

## 七、手动修复流程

```
fraud proof 成立
  │
  ├── 标记 failed node，订单进入 Degraded
  ├── 按最低罚没额/比例/重复违规系数罚没 USDT
  ├── 用户手动选择替补节点
  ├── 替补 join(topic)，从存活节点或临时种子全量拷贝完整文件
  ├── 替补提交 chunkMerkleRoot 确认
  └── 替补加入 order.nodes[]
```

v1 不做自动 repair 调度和自动带宽费结算。若 topic 内没有可用源，订单保持 `Degraded`，继续等待节点恢复或临时种子；超过协议窗口且副本数不足时，进入 `Cancelled` 或提前结算。

## 十、合约安全

| 风险 | 应对 |
| ---- | ---- |
| 重入攻击 | OpenZeppelin ReentrancyGuard |
| 权限滥用 | 分模块角色控制 + 时间锁升级 |
| USDT 脱锚或合规风险 | v1 仅支持 USDT 以降低复杂度；后续如有需要，可通过协议升级扩展其他结算资产 |
| 诬告攻击 | 提交者质押 + counter-proof |
| 报名女巫 | USDT 质押、容量上限、信誉分 |
| 参数滥用 | 多签 + 时间锁 + 参数变更公开记录 |
