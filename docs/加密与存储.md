## 一、应用层加密模块

### 1.1 文件加密流程

```
原始文件 → AES-256-GCM 加密 → 加密分片 → Hyperdrive 存储
                 ↓
          加密密钥（用户持有）
                 ↓
          通过 mostWallet 体系派生
```


### 1.4 元数据加密方案

当前 `published-files.json` 明文存储：

```json
{ "cid": "bafy...", "name": "photo.jpg", "size": 1024, "path": "/photos" }
```

改造后：

```json
{
  "cid": "bafy...",
  "encryptedMetadata": "U2FsdGVkX1+...",
  "encryptedKey": "encrypted_with_node_public_key"
}
```

- 文件元数据用文件密钥 AES-256-GCM 加密
- 文件密钥用存储节点的 X25519 公钥加密（仅该节点可解密用于证明）
- 用户保留原始文件密钥用于下载解密

### 1.5 对现有代码的影响

| 文件                     | 修改内容                                                |
| ------------------------ | ------------------------------------------------------- |
| `server/src/index.js`    | `publishFile` 增加加密步骤；`downloadFile` 增加解密步骤 |
| `server/src/core/cid.js` | 保持不变（CID 基于加密后内容计算）                      |
| `app/app/page.tsx`       | 上传/下载 UI 增加加密状态提示                           |

## 二、文件分片与纠删码

### 2.1 分片策略

```
原始文件（100MB）
    ↓
AES-256 加密
    ↓
分片（每片 10MB）→ 10 个分片
    ↓
纠删码编码（10+4）→ 14 个分片（容忍 4 个丢失）
    ↓
分发到不同节点（每节点 1 片）
```

### 2.3 数据修复机制

```javascript
// 节点定期健康检查
async function healthCheck() {
  for (const order of activeOrders) {
    const availableShards = await checkShardAvailability(order)
    if (availableShards < order.redundancy) {
      // 触发修复：从可用分片重建丢失分片
      await repairOrder(order, availableShards)
    }
  }

```

### 2.4 纠删码库选择

| 库              | 语言      | 性能 | 维护状态 |
| --------------- | --------- | ---- | -------- |
| reed-solomon-js | JS        | 中   | 活跃     |
| @noble/hashes   | JS        | 高   | 活跃     |
| WASM RS         | Rust→WASM | 极高 | 活跃     |

**决策**：先用 `reed-solomon-js` 快速验证，性能瓶颈时换 WASM。
