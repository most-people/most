# MostBox 存储与可选加密设计

> 以 `项目计划书.md` 为准：MVP 不内置默认加密，不做分片/纠删码；Hyperdrive 存完整文件副本，隐私文件由用户在本地自行加密后再上传。

---

## 一、MVP 存储模型

```
原始文件
  │
  ├── 计算 UnixFS CID v1
  ├── 按 256KB chunk 构建 Merkle root
  ├── 写入 Hyperdrive：key = /<cid>
  ├── 上传者 join(topic)
  └── 存储节点 pull 完整文件副本
```

| 项目 | 规则 |
| ---- | ---- |
| 文件内容 | 默认明文完整副本 |
| Hyperdrive key | `/<cid>` |
| CID | 现有 UnixFS CID v1 |
| topic | `rootCid.multihash.digest` |
| 副本数 | MVP 默认 `m=2` |
| 校验 | CID + Merkle root |
| 元数据 | `published-files.json` 维护显示路径和文件信息 |

Hyperdrive 只负责存储文件内容。目录结构、显示路径、收藏、回收站等应用级元数据仍由 `published-files.json` 维护，移动/重命名只更新 JSON，不移动 Hyperdrive 内容。

---

## 二、为什么 MVP 不默认加密

项目计划书选择“CID 即权限”的公开内容寻址模型：

- 知道 CID 即可查询订单、定位节点、下载文件、续费。
- 存储节点持有完整文件副本，便于直接下载、做种、repair 和 Merkle 审计。
- 默认加密会引入密钥分发、恢复、分享、节点证明与 UX 复杂度，不进入当前 MVP 主线。

因此，MostBox MVP 的隐私边界是：

| 场景 | 处理方式 |
| ---- | -------- |
| 公共可分享文件 | 直接上传，CID 可自由传播 |
| 私密文件 | 上传者在本地自行加密后，将密文作为普通文件上传 |
| CID 泄露 | 设计上等同于获得下载权限 |
| 文件名/路径隐私 | MVP 不承诺链上或节点侧隐私；前端显示元数据仅用于本地应用体验 |

## 三、可选本地加密流程

MostBox 不负责管理密钥，但允许用户把加密后的文件作为输入：

```
用户本地工具 / 第三方加密器
  │
  ├── 原始文件 → 密文文件
  ├── 用户自行保存密钥或口令
  └── 密文文件进入 MostBox 上传流程
        │
        ├── CID = hash(密文的 UnixFS CID)
        ├── Merkle root 覆盖密文 chunk
        └── 节点存储密文完整副本
```

这意味着 MostBox 对密文没有特殊认知：下载者拿到密文后，必须用上传者另行分享的密钥解密。

## 四、Merkle 校验

文件按固定 chunk 构建一棵 Merkle 树：

```
文件 [chunk_0, chunk_1, ... chunk_n]
  │
  ├── leaf[0] = hash(chunk_0)
  ├── leaf[1] = hash(chunk_1)
  └── merkleRoot 写入链上订单
```

| 参数 | 值 |
| ---- | -- |
| chunk 大小 | 256KB |
| 链上字段 | `merkleRoot`, `chunkSize` |
| 下载校验 | 重算 Merkle root + 重算 CID |
| 审计校验 | 请求指定 chunk + Merkle proof |

## 五、完整副本与修复

MVP 不做分片，也不使用 Reed-Solomon 纠删码。

| 能力 | MVP 决策 |
| ---- | -------- |
| 副本方式 | `m` 个节点各自持有完整文件 |
| 默认副本数 | `m=2` |
| 下载来源 | 任意 1 个持有完整副本的 peer |
| 修复方式 | 从存活节点全量拷贝到替补节点 |
| 热门文件 | 下载者完成后临时做种，分摊质押节点带宽 |

对比旧方案：

| 旧方案 | 当前决策 |
| ------ | -------- |
| `sharding.js` 文件分片 | 暂不实现 |
| Reed-Solomon 纠删码 | 暂不实现 |
| 分片 CID | 暂不实现 |
| 从分片重建丢失数据 | 改为从完整副本全量 repair |

## 六、对现有代码的影响

| 文件 | 修改方向 |
| ---- | -------- |
| `server/src/core/cid.js` | 继续作为 UnixFS CID v1 计算入口 |
| `server/src/index.js` | 增加 Merkle root 构建、topic 下载、临时做种控制 |
| `published-files.json` | 继续维护应用层目录结构和展示元数据 |
| 前端文件管理 | 增加付费存储状态、订单状态、可选“本地加密后上传”提示 |

不新增默认的 `encryption.js` 或 `sharding.js`，除非未来路线重新确认进入 V2。
